身份验证策略是用于验证用户或系统身份以授予对受保护资源的访问权限的方法或技术。可使用多种不同的身份验证策略，包括：


# Basic Authentication


![image.png](images/blog/2023-05-29-99b4050f/img-ee8446bd8a.png)


HTTP基本认证是一种简单的身份验证机制，它允许Web服务器要求用户提供用户名和密码才能访问某些受保护的资源。这种认证方式广泛应用于Web开发中，尤其是在需要保护API或某些敏感页面时。


基本认证是一种通过在HTTP请求头中发送用户名和密码来验证用户访问资源权限的机制。


## 基本认证的工作原理


1. 客户端请求访问受保护的URL：

- 客户端尝试访问一个需要认证的URL。例如`[https://some.url](https://some.url/)`。

2. 服务器检查请求头中的授权信息：

- 服务器检查请求是否包含有效的用户名和密码。
- 如果凭证不存在或无效，服务器会返回401 Unauthorized状态码，提示客户端需要认证。

3. 浏览器提示输入凭证：

- 浏览器检测到响应中的`www-authenticate`头，并提示用户输入用户名和密码。
- 用户输入凭证后，浏览器会将这些凭证进行Base64编码。

4. 用户提交凭证：

- 用户提交用户名和密码后，浏览器将它们编码为Base64格式，并在下一次请求的`Authorization`头中发送。

5. 服务器继续认证循环：

- 服务器接收到新的请求后，再次进行认证检查。如果凭证有效，服务器允许访问；如果无效，继续返回401 Unauthorized状态码。

## 认证过程中的关键概念

- Authorization Header：请求头中的`Authorization`字段用于发送认证信息。
- WWW-Authenticate Header：响应头中的`www-Authenticate`字段用于指示客户端需要哪种类型的认证。
- Base64 Encoding：一种编码方式，用于将用户名和密码转换为可以安全传输的格式。
- Realm：保护空间的概念，指的是一组使用相同认证信息的页面或API。

## 安全性考虑

- 不安全性：基本认证的凭证虽然经过Base64编码，但这种编码是可逆的，因此如果通信不被加密，任何人都可以解码这些凭证。
- TLS/HTTPS：为了提高安全性，基本认证应该在TLS/HTTPS上使用，这样凭证在传输过程中就会被加密，防止被截获和解码。

## 基本认证的限制和替代方案

- 限制：基本认证不支持复杂的认证场景，如多因素认证或基于角色的访问控制。
- 替代方案：对于需要更高级认证功能的场景，可以使用OAuth、JWT（JSON Web Tokens）等更复杂的认证机制。
> 注意事项：
>
> 基本认证不是安全的认证方式，除非它与TLS/HTTPS一起使用。
>
>
> 因为任何人都可以截获并解码Base64编码的凭证。
>
>

# Session Based Authentication


![image.png](images/blog/2023-05-29-99b4050f/img-4e4fd0ea65.png)


## 会话认证概述


会话认证是一种用于Web应用程序的身份验证技术，它通过服务器端存储的会话信息来识别用户。与无状态认证（如基于令牌的认证）不同，会话认证需要服务器记住每个用户的状态。


## 有状态 vs 无状态认证方法

- 有状态认证：
    - 服务器跟踪已登录用户的状态，并可以在内存或存储中存储会话信息。
    - 可以撤销认证会话。
- 无状态认证：
    - 服务器不跟踪用户状态，通常使用令牌来验证用户。
    - 一旦发出，不能简单地撤销认证会话。

## 会话认证的工作原理


1. 客户端发送登录请求：

- 用户通过浏览器向服务器发送登录请求，通常包含用户名和密码。

2. 服务器验证凭证并创建会话：

- 服务器验证用户名和密码。
- 如果凭证有效，服务器创建一个新的会话，并生成一个唯一的会话标识符（Session ID）。
- 服务器将这个会话ID存储在内存或某种形式的存储中，并将其与用户关联起来。

3. 服务器返回会话ID：

- 服务器将生成的会话ID发送回客户端。
- 客户端接收会话ID并将其存储在Cookie中（如果启用了Cookie），或者存储在本地或会话存储中（如果未启用Cookie）。

4. 客户端发送带有会话ID的请求：

- 客户端在随后的所有请求中都包含这个会话ID。
- 服务器检查存储中是否存在与当前会话ID关联的用户。
- 如果存在，服务器允许访问请求的资源；如果不存在或会话ID无效，服务器返回401 Unauthorized响应。

5. 用户登出时销毁会话：

- 当用户选择登出时，会话在服务器端被销毁，并且客户端的会话ID（如Cookie）也被清除。
- 相同的会话ID不能被再次使用。

## 会话认证的关键点

- 会话ID：一个随机生成的唯一标识符，用于在客户端和服务器之间标识用户的会话。
- 存储会话：服务器可以在内存、数据库、文件系统等地方存储会话信息。
- 安全性：会话ID应该足够随机，以防止预测攻击。此外，使用HTTPS可以保护会话ID在传输过程中的安全。
- Cookie：一种存储会话ID的方式，可以设置为HttpOnly和Secure，以提高安全性。
- 登出：用户登出时，会话应该被服务器端销毁，以确保会话ID不再有效。

## 会话认证的优缺点

- 优点：
    - 用户体验好，因为用户在会话期间不需要重复登录。
    - 服务器可以跟踪用户状态，提供个性化服务。
- 缺点：
    - 需要服务器存储会话信息，可能会增加服务器的存储和计算负担。
    - 如果不妥善管理会话ID，可能会面临安全风险。

# Token Based Authentication


![image.png](images/blog/2023-05-29-99b4050f/img-ff66c1b55d.png)


## 令牌认证概述


令牌认证是一种无状态的认证方法，它使用令牌（Token）来验证客户端的请求。与基本认证和会话认证不同，令牌认证不需要在服务器端存储用户状态。


## 令牌认证的工作原理


1. 客户端发送凭证以生成令牌：

- 用户通过浏览器向服务器发送用户名和密码。

2. 服务器验证凭证：

- 服务器接收凭证并进行验证。
- 如果凭证有效，服务器生成一个令牌并将其发送给客户端。
- 如果凭证无效，服务器返回一个错误响应，例如422 Unprocessable Entity。

3. 客户端存储令牌：

- 客户端接收到令牌后，将其存储在本地存储或Cookie中。

4. 客户端发送带有令牌的请求：

- 在随后的所有请求中，客户端都在请求头中包含这个令牌。

5. 服务器验证令牌：

- 服务器检查每个请求中的令牌。
- 如果令牌有效，服务器处理请求并返回响应。
- 如果令牌无效，服务器返回401 Unauthorized错误。

## 令牌的特点

- 随机字符串：令牌通常是一个随机生成的字符串，看起来没有明显的规律。
- 无状态：服务器不存储令牌，因此令牌认证是无状态的。
- 过期时间：令牌有一个有效期，过期后令牌将不再有效。
- 签名：令牌通常使用密钥签名，以防止篡改，确保服务器可以信任令牌。
- 授权头：令牌通常在`Authorization`头中发送。
- 不透明或自包含：
    - 不透明令牌（Opaque Token）：令牌本身不包含任何有意义的数据，只能由授权服务器验证，类似于会话ID。
    - 自包含令牌（Self-Contained Token）：令牌本身包含了数据，可以由客户端查看，例如JWT（JSON Web Tokens）。

## 令牌认证的策略示例

- SWT（Simple Web Tokens）：简单Web令牌，一种基于令牌的认证机制。
- JWT（JSON Web Tokens）：JSON Web令牌，一种自包含的令牌，包含了用户信息和签名。
- OAuth（Open Authorization）：开放授权，一种允许应用程序通过第三方服务获取有限权限的框架。
- SAML（Security Assertions Markup Language）：安全断言标记语言，一种基于XML的令牌标准，用于身份验证和授权。
- OpenID：一种用于用户身份验证的开放标准。

## 令牌认证的优缺点

- 优点：
    - 无状态，易于扩展。
    - 令牌可以在多个服务之间共享，便于实现单点登录（SSO）。
    - 可以通过设置较短的过期时间来提高安全性。
- 缺点：
    - 需要在客户端安全地存储令牌。
    - 如果令牌被盗用，可能会危及用户账户的安全。

# JWT Authentication


![image.png](images/blog/2023-05-29-99b4050f/img-bc4f487325.png)


JWT（JSON Web Tokens）是一种用于在网络应用环境间传递声明（claims）的方式，它以JSON对象的形式进行编码，并通过数字签名确保其内容不被篡改。一种基于令牌的认证方法，遵循开放标准RFC 7519。


## JWT的工作原理


1. 验证凭证：

- 用户提交用户名和密码到认证服务器。

2. 生成令牌：

- 如果凭证有效，服务器生成一个JWT并发送给客户端。

3. 客户端存储令牌：

- 客户端将JWT存储在本地存储、Cookie或请求头中。

4. 客户端发送请求：

- 客户端在随后的请求中将JWT作为认证信息发送给服务器。

5. 服务器验证JWT：

- 服务器验证JWT的签名和有效性，如果JWT有效，服务器处理请求并返回响应。

## JWT的特点

- 自包含：JWT携带了所有必要的用户信息和声明（claims），因此它是自描述的。
- URL安全：JWT是URL安全的字符串，可以在HTTP请求中安全地传递。
- 存储在客户端：JWT通常存储在客户端的本地存储或Cookie中。

## JWT的结构


JWT由三部分组成，用点（.）分隔：


1. Header（头部）：

- 包含令牌的类型（这里是JWT）和所使用的签名算法（如HS256）。

2. Payload（负载）：

- 包含所谓的声明，可以是注册声明、公开声明和私有声明。

3. Signature（签名）：

- 使用Base64对Header和Payload进行编码后，使用服务器端的密钥进行签名。

## JWT的声明（Claims）

- 注册声明（Registered Claims）：
    - 标准名称，用于JWT的预定义用途，如：
        - `iat`：签发时间。
        - `iss`：签发者。
        - `sub`：主题。
        - `exp`：过期时间。
        - `aud`：受众。
        - `nbf`：生效时间。
        - `jti`：JWT的唯一标识符。
- 公开声明（Public Claims）：
    - 可以自定义的声明，用于存储额外信息，如用户ID、电子邮件等。
- 私有声明（Private Claims）：
    - 只有令牌的接收者和发行者理解的自定义声明。

## JWT的安全性

- 签名：JWT的签名部分确保了令牌内容的完整性和真实性。
- 加密：虽然JWT可以被Base64解码查看内容，但不应包含敏感信息，或应通过HTTPS等加密方式传输。

## JWT的使用场景

- 身份验证：用户登录后，服务器生成JWT，用户在后续请求中使用JWT进行身份验证。
- 信息交换：在不同服务或微服务架构中，JWT可以用于安全地交换信息。

## JWT的优缺点

- 优点：
    - 无状态和可扩展性，适合分布式系统。
    - 携带所有必要信息，减少了服务器查询数据库的需求。
    - 支持跨域认证。
- 缺点：
    - 如果不使用HTTPS，JWT内容可以被拦截和篡改。
    - JWT一旦发出，不能主动废止，除非使用额外的机制（如黑名单）。

# OAuth


![image.png](images/blog/2023-05-29-99b4050f/img-c94dff74e8.png)


## 静态介绍

- OAuth — Open Authorization:
    - 定义：一种开放的授权协议，允许用户共享其资源给第三方。
    - 概念：用户授权第三方应用访问其资源，而无需分享登录凭证。

## OAuth的两个版本

- OAuth 1.0：较复杂，如今较少使用。
- OAuth 2.0：当前广泛应用的版本。

## 工作原理


OAuth 2.0主要通过四种授权许可来分发令牌。


### 四种授权许可类型


1. 授权码许可（Authorization Code Grant Flow）：

- 主要用于服务器端应用。
- 安全性高，因为它通过服务器端的客户端密钥进行更安全的通信。

2. 隐式许可（Implicit Grant Flow）：

- 主要用于单页面应用和移动应用。
- 较简化，但安全性稍低，因为它直接返回令牌给客户端，而无需服务器端交互。

3. 客户端凭证许可（Client Credentials Grant Flow）：

- 一般用于应用程序之间的通信，或在没有用户交互的情况下请求资源。
- 通过客户端ID和客户端密钥直接获取访问令牌。

4. 资源所有者密码凭证许可（Resource Owner Password Credentials Grant Flow）：

- 用户直接提供其用户名和密码。
- 通常情况下不推荐使用，因为需要直接暴露用户的凭证给客户端。

## 授权码许可流程详细步骤：


1. 用户请求访问资源：

- 用户点击应用中的登录按钮。

2. 用户被重定向到授权服务器：

- 应用将用户重定向到授权服务器，并附带客户端ID和回调URL（redirect_uri）。

3. 用户在授权服务器上进行身份验证和授权：

- 用户输入其凭证（如用户名、密码），并授权应用访问其资源。

4. 授权服务器返回授权码：

- 用户被重定向回应用的回调URL，并附带授权码作为查询参数。

5. 应用向授权服务器请求访问令牌：

- 应用使用授权码、客户端ID、客户端密钥（如果有）向授权服务器请求访问令牌。

6. 授权服务器返回访问令牌：

- 包括access_token（访问令牌）和可能的refresh_token（刷新令牌）。

7. 应用使用访问令牌请求资源：

- 应用使用获取的访问令牌请求所需资源。

8. 资源服务器返回资源数据：

- 验证访问令牌的合法性后，资源服务器返回相应的数据给应用。
> 一些授权流程可以选择性地使用刷新令牌（refresh tokens）来获取新的访问令牌，而无需重新授权。
> - 使用隐式许可可能会带来Token泄露的风险，适用于对安全要求相对较低的情况。
> - 使用客户端凭证许可时，要求客户端具备良好的安全保障措施。
> - 每种授权许可都有其特定的使用场景和安全考虑，选择时应根据实际需求和安全性来衡量。
>

# SSO


![image.png](images/blog/2023-05-29-99b4050f/img-b741fce5f9.png)


单点登录是一种认证策略，允许用户使用单一的用户名和密码登录多个相关但独立的服务。SSO提高了用户体验，减少了记忆多个凭证的需要，并简化了访问多个服务的过程。


## SSO的例子


Google服务：用户可以使用Gmail账户登录Google Meet、YouTube、Google Calendar和Google Docs等所有Google产品。


## SAML（Security Assertion Markup Language）


SAML是一个开放标准，用于在参与方之间交换认证和授权数据。它在SSO实现中扮演着重要角色。


## SAML的三个参与方


1. 用户（User）：需要访问资源的一方。


2. 身份提供者（Identity Provider, IDP）：负责认证用户身份并提供关于用户身份和权限的信息。


3. 服务提供者（Service Provider, SP）：提供用户需要访问的服务。


## SAML的工作流程


1. 用户尝试访问SP控制的资源。


2. SP识别用户未认证，并将用户重定向到IDP。


3. 用户在IDP处进行认证。


4. IDP生成一个包含用户身份和权限信息的SAML断言。


5. IDP将SAML断言发送给SP。


6. SP验证SAML断言的有效性，并根据断言中的信息提供对资源的访问。


## SAML断言

- SAML断言是一个XML文档，包含授权信息。
- SAML断言格式和内容由SP和IDP协商确定。
- SAML断言通常被签名，以确保数据的完整性和来源可信。

## SAML元数据

- 包含配置信息和证书的XML元数据文件放置在IDP和SP。
- 配置信息可能包括需要在SAML断言中包含电子邮件地址等要求。
- 证书用于验证断言签名的有效性，确保数据可信。

## SSO的两种主要认证流程


1. IDP初始化流程（Identity Provider Initiated Flow）：

- 客户端直接请求IDP启动认证流程。

2. SP初始化流程（Service Provider Initiated Flow）：

- 第一个请求发送到SP，然后SP引导用户到IDP进行认证。

## 简化的SSO工作流程示例

- 用户请求访问SP1的资源。
- SP1识别用户未认证，重定向用户到IDP。
- 用户在IDP处输入凭证进行认证。
- IDP验证凭证，生成SAML断言，并将其发送给SP1。
- SP1验证SAML断言，如果有效，为用户提供资源访问权限。

## SSO的优势

- 用户体验：用户只需记住一组凭证。
- 管理效率：集中管理用户账户和权限。
- 安全性：减少凭证泄露的风险，通过集中认证点控制访问。

## SSO的挑战

- 复杂性：SAML和SSO的实现可能相当复杂。
- 信任关系：需要在IDP和SP之间建立信任关系。
- 性能：认证流程可能影响系统性能。
