# **1. 仅依赖 Create React App （CRA） 而不探索现代替代方案**


多年来，Create React App （CRA） 一直是快速设置 React 项目的首选工具。虽然它仍然是一个可行的选择，但盲目地使用 CRA 而不考虑替代方案可能意味着错过显着的性能改进和开发人员体验增强。


**为什么 CRA 可能不是目前的最佳选择：**

- 与较新的工具相比，构建时间较慢
- 配置灵活性较低
- 可能存在过时的依赖项

**可考虑的现代替代方案：**

- Vite：提供闪电般快速的热模块更换和构建时间
- Next.js：提供服务器端渲染和出色的开发人员体验
- Remix：一个全栈式 React 框架，专注于 Web 标准和性能

**如何选择：**

- 评估项目要求（SSR、静态站点生成等）
- 考虑团队对不同工具的熟悉程度
- 评估每个方案的社区支持和生态系统
> 尝试使用不同的工具构建一个小型项目，以了解其工作流程和性能特征。

# **2. 不必要地在每个文件中导入 React**


从 React 17 开始，JSX 转换已经更新，无需在每个使用 JSX 的文件中导入 React。此更改可能看起来很小，但它可能会对代码库的整洁性和可维护性产生重大影响。


**跳过不必要的 React 导入的好处：**

- 减少样板代码
- 更清晰的文件头
- 捆绑包大小略有改善

**Before (pre-React 17):**


`import React from 'react';

function MyComponent() {
  return <div>Hello, World!</div>;
}`


**After (React 17+):**


`function MyComponent() {
  return <div>Hello, World!</div>;
}`

> 如果你正在使用 useState 或 useEffect 等 React 方法，你仍然需要显式导入它们：
>
> `import { useState, useEffect } from 'react';
>
> function MyComponent() {
>   const [count, setCount] = useState(0);
>   // ...
> }`
>
>

# **3. 坚持使用 PropTypes 而不是拥抱 TypeScript**


PropTypes 很好地为 React 开发人员提供了类型检查服务，但随着项目复杂性的增加，TypeScript 提供了更健壮的解决方案来确保类型安全和改善整体开发体验。


**为什么 TypeScript 胜过 PropTypes：**

- 在编译时而不是运行时捕获错误
- 通过更好的自动完成和重构工具改进了 IDE 支持
- 更易于理解和维护的自文档化代码

**比较一下 PropTypes 和 TypeScript：**


**使用 PropTypes：**


`import PropTypes from 'prop-types';

function UserProfile({ name, age, email }) {
  return (
    <div>
      <h1>{name}</h1>
      <p>Age: {age}</p>
      <p>Email: {email}</p>
    </div>
  );
}

UserProfile.propTypes = {
  name: PropTypes.string.isRequired,
  age: PropTypes.number,
  email: PropTypes.string
};`


**使用 TypeScript：**


`interface UserProfileProps {
  name: string;
  age?: number;
  email?: string;
}

function UserProfile({ name, age, email }: UserProfileProps) {
  return (
    <div>
      <h1>{name}</h1>
      <p>Age: {age}</p>
      <p>Email: {email}</p>
    </div>
  );
}`


TypeScript 版本不仅提供了更好的类型检查，而且还可以作为组件 props 的清晰文档。

> 迁移提示：如果有正在处理现有项目，则可以通过将文件重命名为 .tsx 并逐步添加类型注释来逐步采用 TypeScript。

# **4. 使用 defaultProps 作为默认值而不是 ES6 解构**


defaultProps 模式在 React 中被广泛使用，但随着功能组件和 ES6 功能的出现，有一种更简洁的方式来为 props 提供默认值。


**为什么首选 ES6 解构：**

- 将所有与 prop 相关的逻辑保存在一个地方
- 更直观，符合现代 JavaScript 实践
- 与函数式组件和类组件无缝协作

**比较一下这俩方法：**


**使用 defaultProps：**


`function Button({ text, color }) {
  return <button style={{ color }}>{text}</button>;
}

Button.defaultProps = {
  text: 'Click me',
  color: 'blue'
};`


**使用 ES6 解构：**


`function Button({ text = 'Click me', color = 'blue' }) {
  return <button style={{ color }}>{text}</button>;
}`


ES6 解构方法更简洁，并将组件的所有 prop 逻辑放在一起，使其更容易一目了然。

> 可以将其与 TypeScript 结合使用，以获得更好的 prop 管理：
>
> `interface ButtonProps {
>   text?: string;
>   color?: string;
> }
>
> function Button({ text = 'Click me', color = 'blue' }: ButtonProps) {
>   return <button style={{ color }}>{text}</button>;
> }`
>
>

# **5. 选择 class 组件而不是功能组件和 hook**


虽然 class 组件多年来一直是 React 的基本组成部分，但函数式组件与 hook 相结合已成为当前编写 React 代码的首选模式。


**功能组件和Hook的优点：**

- 语法更简单，更易于阅读
- 由于减少了开销，在某些情况下性能更好
- 更易于跨组件测试和重用逻辑
- 更符合现代 JavaScript 实践

将 class 组件重构为带有 hook 的函数式组件：


**Class Component:**


`class Counter extends React.Component {
  constructor(props) {
    super(props);
    this.state = { count: 0 };
  }

  componentDidMount() {
    document.title = `Count: ${this.state.count}`;
  }

  componentDidUpdate() {
    document.title = `Count: ${this.state.count}`;
  }

  increment = () => {
    this.setState(prevState => ({ count: prevState.count + 1 }));
  }

  render() {
    return (
      <div>
        <p>Count: {this.state.count}</p>
        <button onClick={this.increment}>Increment</button>
      </div>
    );
  }
}`


**带 Hook 的功能组件：**


`import { useState, useEffect } from 'react';

function Counter() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    document.title = `Count: ${count}`;
  }, [count]);

  const increment = () => setCount(prevCount => prevCount + 1);

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={increment}>Increment</button>
    </div>
  );
}`


功能组件版本更简洁，更容易理解，并使用现代 React 模式，如 useState 和 useEffect 钩子。
