## **流式传输**


流式响应允许人工智能系统在计算响应的同时，逐步生成并显示输出。系统无需等待整个响应生成完毕即可显示，而是在数据准备就绪后，分批发送较小的数据块（令牌）。这种功能使交互更加流畅自然，如同进行真实的对话。


## **流式响应的工作原理**


### **1. Token-by-Token Generation**


LLM一次生成一个词元（token）。词元是文本的一个单元，它可以是：

- 一个词（例如，“快乐”）
- 单词的一部分（例如，“happiness”中的“happ-”）
- 单个字符（例如，“a”或标点符号，如“,”）。

当用户提交查询时，LLM 开始按顺序生成令牌。第一个令牌生成后，立即发送到客户端接口，然后该过程持续进行，直到完成整个响应。这种令牌的增量交付方式构成了流式响应的基础。


### **2. 流式 API**


大多数 LLM 都支持通过专用 API 进行流式传输。例如，OpenAI 的 API 包含一个 `stream` 参数，允许客户端接收实时令牌流，而无需等待完整的响应。以下是其工作原理：

- **步骤 1：** 客户端向启用流式传输的服务器发送查询。
- **步骤 2：** 服务器处理输入并开始生成令牌。
- **步骤 3：** 令牌准备就绪后，会分小块逐个发送给客户端。
- **步骤 4：** 客户端实时地将每个数据块添加到显示中。

这会让用户产生人工智能“输入”回复的错觉。


### **3. 客户端实时渲染**


在客户端，应用程序被设计成立即渲染接收到的令牌或数据块。例如：

- Web 应用程序可能会在收到新令牌后立即更新用户界面。
- 基于终端的程序可能会将每个标记直接刷新到输出流中，从而实现实时“输入”效果。

## **实现流式的关键技术**


### **a)** Server-Sent Events (SSE)


服务器发送事件 (SSE) 是一种协议，它允许服务器通过单个 HTTP 连接实时向客户端推送更新。每个数据块都作为单独的事件发送。

每个 `data` 字段代表客户端可以立即显示的响应的一部分。


### **b) WebSocket**


WebSocket 提供客户端和服务器之间的双向通信通道，这对于流式传输尤为有用。虽然 WebSocket 在简单的文本流传输中并不常见，但它经常用于更复杂的实时应用程序，例如协作编辑器或实时仪表板。


### **c) 异步编程**


异步编程框架，例如 Python 的 `asyncio` 或 JavaScript 的 `Node.js` 对于高效处理流式传输至关重要。这些框架使服务器能够：

- 同时处理多个客户端请求。
- 向客户端发送令牌，而不阻塞后续令牌的生成。

## **Pipeline Optimization in LLMs**


LLM 中高度优化的架构和解码策略使得流式传输成为可能：


### **1. Transformer 架构**


LLM（层级模型）采用 Transformer 架构，该架构并行处理输入，但顺序生成输出。每个词元都基于前面词元的上下文进行预测，从而实现流畅的生成流程。


###  **2. 解码策略**

- **Beam Search:** 生成多个潜在序列并选择最有可能的序列。
- **Sampling:** I引入随机性以产生多样化的响应。
- **Top-k Sampling or Nucleus Sampling:** 通过将标记选择限制在最有可能的候选者中来平衡质量和创造力。

## **流式响应的优势**


**1. 更快的感知反应时间**


**2. 增强的交互性**


**3. 高效的资源利用**


## **流式响应的挑战**

- **网络延迟：** 连接速度慢或连接不稳定会影响实时体验。
- **错误处理：** 确保从中断或令牌生成失败中优雅恢复需要精心实现。
- **复杂性：** 实现流式响应会增加服务器端和客户端代码的复杂性。
