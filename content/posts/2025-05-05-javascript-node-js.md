在许多标准的企业应用中，一致的日志记录有多种用途。它帮助企业识别和纠正错误，提供有价值的分析见解，并让你测试新解决方案。然而，这也使得日志注入成为黑客劫持或甚至访问敏感用户信息的最常见方式之一。应用程序记录数据的方式中的漏洞可能允许攻击者将恶意代码注入日志，从而危及应用程序的完整性和机密性。


# 日志注入漏洞


当涉及到日志信息时，某些应用程序会记录到外部文件或依赖于发送日志的外部 API。当攻击者操纵输入数据以将恶意代码注入应用程序日志时，就会发生日志注入。这些日志通常用于调试、监视和审核目的。


当发生日志注入时，攻击者可以使日志混乱，甚至在应用程序的上下文中执行代码，从而导致各种安全风险。


## **易受攻击的 Node.js 应用程序示例**


考虑一个简单的 Node.js 应用程序，该应用程序使用 Express Web 框架来记录用户提供的数据，但没有进行适当的验证或清理。以下是展示这种漏洞的代码片段：


```javascript
const express = require('express');
const app = express();

app.get('/log', (req, res) => {
  const userInput = req.query.data;
  console.log('INFO: ' + userInput);
  res.send('Data logged successfully.');
});

app.listen(3000, () => {
  console.log('Server started on port 3000');
});
```


一旦这个服务器在本地启动并运行，你可以使用像 Insomnia 这样的 API 测试工具发起 GET 请求，以测试是否一切如预期运作。在这种情况下，你计划从一个表单中记录一个用户名，以测试服务器日志的情况。你添加符合服务器预期的数据并发送请求。这样做会给你一个 200 OK 状态和一些文本，显示“数据记录成功”。


如果你检查服务器端的日志，一切看起来都按预期记录。由于你将值charlie添加到数据参数中，因此日志显示INFO: charlie正在记录到服务器：


恶意人员滥用该应用程序记录数据方式的一种方法是通过数据参数插入虚假日志。在此示例中，恶意人员只需向查询中添加额外的文本即可：


例如，一个恶意的人可以添加"INFO: User deleted"作为一个虚假日志。当开发团队看到这个日志时，他们可能会被误导，并花费时间和资源试图弄清楚为什么这个应用程序在用户提交用户名时会删除用户，而这一切只是因为攻击者决定在用户输入表单中插入额外的字符。


虽然这个例子很简单，但它有助于你理解日志注入的基本概念。其他类型的注入可能包括通过修改日志消息来添加虚假的错误日志或虚假的用户信息日志，甚至更糟的是插入命令注入，从而使攻击者能够直接操控内部数据。


## **防止日志注入漏洞的方法**


## **清理用户输入**


在大多数情况下，用户输入应在记录之前进行验证和清理。这意味着要确保将要记录的内容已经过彻底检查，以排除会被误认为另一个日志输入的字符和/或只包含应该出现在日志中的字符。


使用之前的例子，一种清理日志的方法是加入一个正则表达式，该表达式只过滤并允许字母，这样可以防止特殊字符或数字被添加到日志中。


```javascript
app.get('/log', (req, res) => {
  const userInput = req.query.data
  const sanitizedInput = userInput.replace(/[^a-zA-Z]+/, '')
  console.log('INFO: Username: ', sanitizedInput);
  res.send('Data logged successfully.');
});
```


需要注意的是，选择正确的数据清理方法可能会很棘手。在这个例子中，用户名中带有数字（例如123）的人，他们的用户名将在日志中被修改。


虽然清理输入可以帮助捕获恶意日志，但这种权衡要求开发人员对过滤掉的内容和执行过滤的方式格外谨慎。例如，错误地编写正则表达式可能会导致 ReDOS 漏洞，从而导致 Node.js 应用程序瘫痪。使用[<u>**validator.js**</u>](https://www.npmjs.com/package/validator)等第三方 npm 包是避免重新发明轮子的好选择，但请确保使这些依赖项保持最新。


## **小心你记录的内容**


每个应用程序都是独一无二的。一种日志记录标准无法满足每个应用程序可能出现的所有潜在需求。重要的是要考虑需要记录哪些信息以及为谁记录。


DEBUG级别的日志可以帮助开发人员追踪其应用程序中的情况，以及用户如何从一个组件导航到下一个组件或信息如何从一个来源流向另一个来源。WARN 或 ERROR 级别的日志可以帮助工程团队及早发现潜在的错误，并了解导致应用程序崩溃或故障的原因。关键是要只记录你需要的内容，不多不少，同时考虑如果日志被攻击者入侵会发生什么。


正如这篇Snyk关于日志漏洞[日志漏洞](https://learn.snyk.io/lesson/logging-vulnerabilities/?_gl=1*1o3iv4d*_gcl_au*MTAxNzA2MTg1NS4xNzMwMDI2OTUy*_ga*OTk2OTIyMTM2LjE3MzAwMjY5NTA.*_ga_X9SH3KP7B4*MTczMDIwMTg5MS4zLjAuMTczMDIwMTg5MS4wLjAuMA..)的文章所解释的那样，向外部日志服务发送敏感信息可能会导致数据被攻击者泄露。通常来说，避免记录敏感或机密信息是个好的做法，比如密码、API密钥或者个人数据。如果你确实需要记录这些信息，一定要实施适当的访问控制，并考虑使用环境变量（例如在GitHub Actions上）和密钥管理系统来安全地管理敏感配置数据。


除了考虑你记录什么之外，你还需要考虑如何记录。结构化日志记录可以提供一种更有条理的方式来记录数据，这样开发人员更容易识别。例如，根据前面的例子，与其直接将用户数据记录为字符串，不如使用一个JSON对象来记录，这样可以以更清晰的方式标记记录的内容：


```javascript
app.get('/log', (req, res) => {
  const userInput = req.query.data;
  const log = {username: userInput, 'action':'submitted username', 'time': new Date()}
  console.log(log)
  res.send('Data logged successfully.');
});
```


更重要的是，以这种方式构建日志可以使攻击者更难注入日志，因为日志对象中有许多属性（例如操作或时间）需要攻击者同时篡改，而不这样做可能会暴露出攻击者试图执行的操作。


虽然 JSON 对象的日志记录是一个简单的示例，但有一些日志记录可以提供结构化日志记录，作为常用console.log有用替代方案。


## **使用日志库而不是 console.log**


许多日志记录库比 console.log 提供了更好的控制和安全性，因为它们通常包括内置的清理和过滤功能，以防止日志注入攻击。它们还为开发人员提供了标志，用于控制哪些日志显示在哪个环境中（测试与生产），以及关于日志应被发送到哪里（本地文件、数据库和/或第三方 API）的灵活性，并提供了配置日志的方法，使其更容易追踪。


我们推荐的一个这样的库是 pino，该库维护良好，整体是一个稳定的包，没有已知的安全漏洞。


Pino经过高性能优化，可以设置在每条日志条目中包含时间戳，支持结构化日志记录和标准系统日志级别。这种额外的复杂性和结构可以增强安全性，从而降低日志被攻击者破坏的可能性。
