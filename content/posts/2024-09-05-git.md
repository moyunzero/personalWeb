# 1. 进行空提交


你是否曾发现自己在README中进行了一些小的调整，以便启动CI构建（或其他集成），并尝试调试一个问题？我曾经经常这样做，直到我了解到以下有用的命令：


```javascript
git commit --allow empty -m 'it works!'
```


这个 git 标志允许触发没有内容的提交，跳过你通常会看到的表示你没有任何内容的错误。


这个技巧对于启动工作流程特别有用，而无需对自述文件或其他文件进行微小的更改。 git diff将不会显示任何更改，但是当你需要启动 CI 运行甚至生产部署而无需推送任意代码时，提交会很有帮助。


# 2. 让你的日志变得漂亮！


运行git log是查看代码库最近更改的一种非常有用的方法，但如果你在具有大量分支的项目上运行它，你的眼睛很快就会迷失。


另一个 git 技巧是让git log更有趣，添加一点颜色来帮助你的眼睛和大脑更好地阅读屏幕上的内容。


要获得更好的git log而不需要对 Git 配置进行任何更改，只需运行：


```javascript
git log --pretty=oneline --graph --decorate --all
```


![bad26b9575e204662532b22aba8083b1316373.png](images/blog/2024-09-05-git/img-4e8c3405f9.png)


你会注意到颜色编码和 | 的巧妙使用。和 \ 表示分支。尽管这些标志在功能上不会改变日志中的任何内容，但它们确实使理解git log的输出变得更加容易。


# 3.整理本地分支


随着时间的推移，可能会向 Git 配置添加不同的内容来定制你的体验。


可以使用git config设置，该设置会当执行fetch或pull操作时，可以删除任何已从远程仓库中删除的本地分支。


可以通过运行以下命令快速设置此设置：


```javascript
git config --global fetch.prune true
```


除了更改 Git 配置之外，另一个 Git 技巧是更进一步，通过运行以下 Git 命令删除已合并到 master 分支的本地分支：


```javascript
git branch --merged master | grep -v "master" | xargs -n 1 git branch -d
```


# 4. 使用 Git reflog 恢复已删除的提交


使用git rebase非常有价值，尤其是在使用--interactive标志运行时。尽管如此，变基可能是危险的。当在本地进行变基时，必须强制推送以覆盖远程存储库上的更改。如果经常进行 rebase，最终会意外地 rebase 掉一个（或多个）提交。


只要提交了你的工作，无论使用git rebase做了什么，它仍然存在于本地工作副本中。使用git reflog ，可以找到你需要的 SHA1。然后，运行git checkout <SHA1> ，复制所需内容，然后运行git checkout HEAD返回到本地分支中的最新提交。


# 5. 别名 Git


每隔一段时间，就会出现一个你无法自己解决的错误。有时，想要/需要联系编写似乎不起作用的代码的人。甚至可能只是想与作者交谈以更好地了解代码背后的上下文。


通常使用git blame来找出谁写了一行代码。事实上， git blame显示谁编写了一行代码、何时编写以及在什么提交中将其添加到代码库中。许多代码编辑器和 IDE 都将此功能构建到编辑器中，在每行代码旁边显示名称！


这是 Git 的一个被低估的功能，但运行git blame感觉不太好。


实际上，可以使用 Git 配置中的别名来更改 Git 命令，这是这些 Git 技巧中最无私的。要将git blame改为由git investigate调用，请运行以下命令：


```javascript
git config --global alias.investigate blame
```


可以将investigate更改为你想要的任何单词！你的 Git 配置还可以保存其他命令的 Git 别名。


# 6.使用git add -p暂存提交


使用"git add -p"命令可以以交互式的方式暂存要提交的文件内容。


具体来说，此命令会分析当前所处的工作区，将所有修改过的文件划分为小块（或称为"片段"）。然后，命令会逐个显示这些片段，并要求用户选择是将其暂存（将其包含在下一次的提交中）还是忽略（将其排除在下一次的提交之外）。


在显示每个片段时，用户有几个选项：可以选择"yes"将片段暂存，选择"no"将片段忽略，选择"quit"来退出命令并停止暂存操作，或选择其他选项用于更精细的控制。


通过交互式地暂存文件内容，"git add -p"命令提供了一种灵活的方式来选择性地提交修改，可以避免将不必要或错误的更改包含在提交中。


# 7. 分段存储代码


分段-p标志不仅仅用于暂存代码！当你不想git stash所有文件或整个有更改的文件时，它也可以与git stash命令一起使用。


要在工作目录中逐段git stash代码，请运行：


```javascript
git stash -p
```


运行git stash -p时，您将看到与git add -p类似的交互式屏幕，一次逐步引导你完成每个更改。


# 8.自动化git bisect


自动化的Git Bisect是一种Git命令，用于帮助开发人员快速找到引入问题（例如bug）的代码提交。Bisect是二分查找的意思，通过不断切换代码提交版本，将问题范围缩小到最小化，从而可以定位到特定的代码提交引发问题。


自动化的Git Bisect通过自动切换提交版本以及自动运行测试来加快问题定位的速度。开发人员首先指定一个已知为问题的提交版本和一个已知为正常的提交版本，Git Bisect会自动帮助开发人员不断切换版本并运行测试，根据测试结果判断问题是否存在于该版本中。通过不断二分查找，并根据测试结果快速排除问题代码，最终可以定位到引发问题的具体提交代码。


使用自动化的Git Bisect可以极大地简化问题定位的过程，特别是对于复杂的代码库或存在多个提交引发问题的情况。它可以帮助开发人员快速找到问题代码，从而可以更高效地进行修复和调试工作。


# 9.在Git中使用表情符号


虽然这可能是这些 Git 提示和技巧中最没有立即用处的，但在提交消息中使用表情符号是为你和同事的一天增添一些乐趣的简单方法！


如果知道表情符号的名称，可以轻松地将表情符号插入提交消息中。可以参考[Gitmoji](https://gitmoji.dev/)备忘单来选择相关的表情符号，并将其作为文本插入到提交消息中，如下所示：


```javascript
git commit -m ":package: Add a docker container to the app"
```


# 10.参考Git文档


只需运行以下命令即可获得帮助：


```javascript
git help
```


作为 Git CLI 的一部分，基本上可以轻松获得所有 Git 文档！如果对 Git 完全陌生或者只是想复习一下，甚至可以通过运行以下命令来获取教程：


```javascript
git help tutorial
```
