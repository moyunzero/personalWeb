# **考虑一个无密码的过程，而不是电子邮件/密码**


在我们开始实践之前，先简单提一下：在这篇文章中，我们将以经典的邮箱/密码表单为例进行操作。然而，从安全性的角度来看，密码实际上是让用户登录的最糟糕方式，这种方法存在许多众所周知的问题。让我们快速看看其中的几个：

1. [这份关于 Verison 2018 年数据泄露的报告称](https://www.researchgate.net/publication/324455350_2018_Verizon_Data_Breach_Investigations_Report)，超过 70%的人在不同网站上重复使用密码。攻击者可能会在一些网站泄露的数据中找到一个跨网站共享的密码，并使用它来窃取另一个网站上的账户。此外，与下面我们将讨论的无密码选项相比，实施双因素认证作为解决方案将降低用户体验。
2. 用户经常忘记密码，重置密码花费太多时间。[根据 Transmit Security 的这份报告](https://www.transmitsecurity.com/wp-content/uploads/transmit-security-passwordless-report-the-impact-of-passwords-on-your-business.pdf)，55%的消费者因为登录过程过于复杂而停止使用一个网站。

如果您想改进您的登录表单，第一步可以考虑无密码选项。


以下是当前我最喜欢的无密码选项：

1. 请使用安全、精心设计和维护的第三方解决方案，例如[ Auth0 ](https://auth0.com/)或 [Amazon Cognito](https://aws.amazon.com/cn/cognito/)。在我们的多个客户项目中，包括那些具有最高安全要求的项目，我们使用了这些方法之一进行身份验证。
2. [苹果](https://developer.apple.com/cn/videos/play/wwdc2022/10092/)和[谷歌](https://developers.google.com/identity/passkeys?hl=zh-cn)提出的新密码标准。这篇《纽约时报》的演示和[文章](https://www.passkeys.io/)解释了它是如何工作的以及为什么更好。
3. 实现带有登录链接的电子邮件（许多用户无论如何都会在每个登录时使用 `Remember password` 功能）。

您也可以将这些选项组合起来，例如，为使用新浏览器的用户提供通行证，为没有通行证的用户提供登录链接。


# **1. 设置** **`autocomplete`**


```diff
- <input type="email">
+ <input type="email" autocomplete="username">
- <input type="password">
+ <input type="password" autocomplete="current-password">
  <button>Login</button>
```


密码管理器是降低电子邮件/密码表单安全风险的唯一选择（但即使它们也无法解决所有风险）。这就是为什么支持密码管理器很重要的原因。


该 `<input>` 标签具有非常有用的 `autocomplete` 属性。它还允许密码管理器区分登录表单（ `current-password` ）和注册表单（ `new-password` ）。


```diff
<label>
    New password:
-   <input type="password">
+   <input type="password" autocomplete="new-password">
  </label>
  <label>
    Confirm password:
-   <input type="password">
+   <input type="password" autocomplete="new-password">
  </label>
  <button>Sign Up</button>
```


不要设置 `autocomplete="off"` ，如果你不知道你在做什么！这样做可能会降低用户体验。我们只有在需要隐藏非常敏感的数据（如医疗网站上“症状”字段）时才应使用此设置。


# **2. 设置** **`type="email"`**


登录表单中最常见的错误之一是在电子邮件字段中使用 `type="text"` 而不是 `type="email"` 。为什么这个属性很重要？

1. 浏览器将在自动完成弹出窗口中建议用户的电子邮件（即使用户是第一次打开网站）。
2. 在触摸设备上，用户将看到一个专为输入电子邮件而设计的特定、更舒适的键盘。
3. 它支持内置的电子邮件验证。

如果您不喜欢浏览器的内置验证，不要设置 `type="text"` 。相反，使用 `novalidate` 属性。


# **3. 所有可点击项应使用** **`<button>`** **或** **`<a>`** **，而不是** **`<div>`** **或** **`<span>`**


```diff
- Forgot your password? <span>Reset it here</span>.
+ Forgot your password? <a href="/restore">Reset it here</a>.

- <div>Login</div>
+ <button>Login</button>
```


链接会更改当前页面，我们应该为所有链接使用 `<a>` 。按钮不会更改页面的 URL，只会更改当前页面的状态；我们应该为所有按钮使用 `<button>` 。


使用 `<a>` 标签相比 `<span onClick={…}>` 有许多好处。例如，用户可以在新标签页中打开链接或点击前查看新的 URL。


使用 `<button>` 代替 `<div>` ，可以使您的网站更易于访问并改善键盘用户体验：屏幕阅读器会告诉用户他们正在处理按钮；按钮将具有 `:focus` 状态以改善键盘用户体验。


此外，我们应该使用 `<button>` 来实现显示/隐藏密码功能；如果你使用的是仅图标按钮，别忘了使用 `aria-label` ：


```html
<button type="button" aria-label="Show password"><div class="eye-icon"></button>
```


这里还有一个注意事项： `<a>` 和 `<button>` 也适用于开发目的。通过始终使用 `<a>` / `<button>` 用于交互元素，您可以更容易地为所有交互元素创建 CSS 选择器：


```css
button, a {
  cursor: pointer; /* Using pointer is controversial, this is just an example */
}
```


# **4. 将字段包裹并在** **`<form>`** **标签内提交** **`<button>`** **。**


```diff
- <div>
+ <form>
    <label>Email: <input type="email" autocomplete="username"></label>
    <label>Password: <input type="password" autocomplete="current-password"></label>
+   <button>Login</button>
+ <form>
- </div>
- <button>Login</button>
```


请确保将所有表单字段及其提交 `<button>` 包裹在 `<form>` 标签内。仅使用 `Enter` 提交表单仅在字段位于 `<form>` 标签内且只有一个提交按钮时有效。（此外，利用屏幕阅读器的用户将欣赏这种实现提供的更好导航。）


# **5. 避免使用** **`placeholder`** **作为** **`<label>`**


```diff
- <input placeholder="E-mail" type="email" autocomplete="username">
+ <label>
+   E-mail:
+   <input type="email" autocomplete="username">
+ </label>
```


`placeholder` 属性的创建是为了展示一个潜在输入的示例，而不是描述该输入。因此，我强烈建议不要将其用作 `<label>` 标签的替代品。此外，当用户输入数据时，占位符值将被隐藏，并且它们通常也存在对比度问题。


# **6. 将复选框输入包裹在** **`<label>`** **标签内**


```diff
- <input type="checkbox"> I agree with the privacy policy
+ <label>
+   <input type="checkbox"> I agree with the privacy policy
+ </label>
```


默认情况下，复选框输入的大小非常小，因此可检测点击的区域也小。这意味着用户需要更多的时间来精确地将光标放置在需要的位置。但如果复选框输入被包裹在 `<label>` 标签中，那么点击其文本也会改变复选框的值。（另外，请注意，每个单独的复选框输入都需要自己的 `<label>` 标签。）


当这样做时，最好也添加一个清晰的 `:hover` 效果，以向用户显示他们可以点击文本来触发输入：


```css
label:hover {
  background: oklch(0 0 0 / 10%);
}
```


# **7. 添加一个可见的** **`:focus`** **状态**


```diff
- *:focus {
-   outline: none;
- }
+ button:focus-visible, a:focus-visible, input:focus-visible {
+   outline: 5px solid oklch(60% 0.15 252);
+ }
```


# **8. 为屏幕阅读器标记无效字段**


```diff
<input type="email" autocomplete="username"
-         class="invalid">
+         required aria-invalid="true" aria-errormessage="email-error">
  <div id="email-error">Enter a valid email address</div>
```


`aria-invalid` 和 `aria-errormessage` 显示验证错误，供屏幕阅读器用户使用。


另一条注意事项：使用 `required` 属性提醒屏幕阅读器用户关于必填字段也是很不错的。如果你不喜欢 `required` 带来的浏览器内置验证，确保在实现自己的验证时使用 `aria-required` 属性。


# **9. 防止在用户输入过程中进行验证**


```diff
- input.addEventListener('keyup', () => {
-   if (validate(input)) {
-     markValid(input)
-   } else {
-     markInvalid(input)
-   }
- })
+ input.addEventListener('input', () => {
+   if (validate(input)) {
+     markValid(input)
+   }
+ })
+ input.addEventListener('change', () => {
+   if (validate(input)) {
+     markValid(input)
+   } else {
+     markInvalid(input)
+   }
```


我们不想在用户输入数据到表单时，通过错误动画分散或混淆他们的注意力，因此不要在用户完成输入之前显示一个 `Not valid email` 错误。


# **10. 防止表单被重复提交**


```plain text
form.addEventListener('submit', () => {
  submitButton.disabled = true

  // Fix for Firefox. It persists the dynamic disabled state without this hack.
  submitButton.autocomplete = 'off'

  // We are using setTimeout for page-reload submit.
  // For AJAX, use await and try-finally to enable submit the button again.
  setTimeout(() => {
    submitButton.disabled = false
  }, 2000)
})
```


用户经常不小心双击而不是单击。因此，为了防止显示一些服务器错误，在表单提交时禁用按钮更好。


# **11. 使用 AJAX 时，考虑网络延迟和服务器/网络错误**


```diff
form.addEventListener('submit', async () => {
-   await fetch(…)
+   try {
+     showLoader()
+     await fetch(…)
+   } catch (e) {
+     showError(e)
+   } finally {
+     hideLoader()
+   }
  })
```


对于每个 AJAX 请求，我们始终应该考虑两件事：

1. 显示加载状态给用户。在本地开发时您有 0ms 的延迟，但实际用户在收到服务器响应前可能需要几秒钟，因此用户在点击提交按钮后应该看到某种反应。
2. 处理网络和服务器错误。在本地开发中您可能看不到它们，但在生产环境中每个用户都可能遇到来自服务器的 `WiFi is down` 或 `Error 500` 错误；为它们做好准备，并向用户显示一些适当的文本。

注意：对于认证表单，最好通过页面刷新提交表单，这样做更好，因为它会将用户的令牌保存到 `httpOnly` -cookie 中并更新 Web 应用中的所有存储。
