# 概述


Node.js 是一个开源、跨平台的 JavaScript 运行时环境。它是几乎适用于任何类型项目的流行工具！


Node.js 在浏览器外部运行 V8 JavaScript 引擎，这是 Google Chrome 的核心。这使得 Node.js 具有非常高的性能。


Node.js 应用程序在单个进程中运行，无需为每个请求创建新线程。 Node.js 在其标准库中提供了一组异步 I/O 原语，可防止 JavaScript 代码阻塞，并且通常 Node.js 中的库是使用非阻塞范例编写的，这使得阻塞行为成为例外而不是常态。


当 Node.js 执行 I/O 操作（例如从网络读取、访问数据库或文件系统）时，Node.js 不会阻塞线程并浪费 CPU 周期等待，而是会在响应返回时恢复操作。


这使得 Node.js 能够通过单个服务器处理数千个并发连接，而不会引入管理线程并发的负担，而管理线程并发可能是错误的重要来源。


Node.js 具有独特的优势，因为数百万为浏览器编写 JavaScript 的前端开发人员现在除了客户端代码之外还可以编写服务器端代码，而无需学习完全不同的语言。


在 Node.js 中，可以毫无问题地使用新的 ECMAScript 标准，因为您不必等待所有用户更新其浏览器 - 您负责通过更改 Node.js 版本来决定使用哪个 ECMAScript 版本，您还可以通过运行带有标志的 Node.js 来启用特定的实验性功能。


## Node.js 应用程序示例


Node.js 最常见的 Hello World 示例是 Web 服务器：


```javascript
const { createServer } = require('node:http');

const hostname = '127.0.0.1';
const port = 3000;

const server = createServer((req, res) => {
  res.statusCode = 200;
  res.setHeader('Content-Type', 'text/plain');
  res.end('Hello World');
});

server.listen(port, hostname, () => {
  console.log(`Server running at http://${hostname}:${port}/`);
});
```

Node.js 拥有出色的标准库，包括一流的网络支持。

服务器设置为侦听指定的端口和主机名。当服务器准备就绪时，将调用回调函数，在本例中通知我们服务器正在运行。

# Node.js 和浏览器之间的区别


浏览器和 Node.js 都使用 JavaScript 作为编程语言。构建在浏览器中运行的应用程序与构建 Node.js 应用程序完全不同。尽管它始终是 JavaScript，但存在一些关键差异，导致体验截然不同。


从广泛使用 JavaScript 的前端开发人员的角度来看，Node.js 应用程序带来了巨大的优势：用单一语言对所有内容（前端和后端）进行编程的舒适度。


在浏览器中，大多数时候您所做的事情是与 DOM 或其他 Web 平台 API（如 Cookie）进行交互。当然，Node.js 中不存在这些。您没有**document** 、 **window**和浏览器提供的所有其他对象。


在浏览器中，我们没有 Node.js 通过其模块提供的所有优秀 API，例如文件系统访问功能。


另一个很大的区别是，在 Node.js 中，您可以控制环境。除非您正在构建任何人都可以在任何地方部署的开源应用程序，否则您知道将在哪个版本的 Node.js 上运行该应用程序。与浏览器环境相比，您无法选择访问者将使用的浏览器，这非常方便。


这意味着您可以编写 Node.js 版本支持的所有现代 ES2015+ JavaScript。由于 JavaScript 的发展速度如此之快，但浏览器的升级速度可能有点慢，有时在网络上您只能使用较旧的 JavaScript / ECMAScript 版本。在将代码发送到浏览器之前，您可以使用 Babel 将代码转换为与 ES5 兼容，但在 Node.js 中，您不需要这样做。


另一个区别是 Node.js 同时支持 CommonJS 和 ES 模块系统（自 Node.js v12 起），而在浏览器中，我们开始看到 ES 模块标准正在实施。


实际上，这意味着您可以在 Node.js 中同时使用**require()**和**import** ，但仅限于在浏览器中**import** 。


# Node.js中开发和生产之间的区别


**Node.js 中的开发和生产之间没有区别**，即，无需应用任何特定设置即可使 Node.js 在生产配置中工作。但是，npm 注册表中的一些库可以识别使用**NODE_ENV**变量并将其默认为**development**设置。始终使用**NODE_ENV=production**集运行 Node.js。


配置应用程序的一种流行方法是使用[十二因素方法](https://12factor.net/)。


## NODE_ENV in Express


在广受欢迎的[express](https://expressjs.com/)框架中，将**NODE_ENV**设置为**production**通常可以确保：

- 日志记录保持在最低限度、必要的水平
- 进行更多的缓存级别以优化性能

这通常是通过执行命令来完成的


```javascript
export NODE_ENV=production
```


在 shell 中，但最好将其放在 shell 配置文件中（例如 Bash shell 的**.bash_profile** ），否则在系统重新启动时该设置将不会保留。


您还可以通过将环境变量添加到应用程序初始化命令中来应用环境变量：


```javascript
NODE_ENV=production node app.js
```


例如，在 Express 应用程序中，您可以使用它为每个环境设置不同的错误处理程序：


```javascript
if (process.env.NODE_ENV === 'development') {
  app.use(express.errorHandler({ dumpExceptions: true, showStack: true }));
}
if (process.env.NODE_ENV === 'production') {
  app.use(express.errorHandler());
}
```


例如，如果**NODE_ENV**未设置为**production** ，则[Express](https://expressjs.com/)使用的模板库[Pug](https://pugjs.org/)将在调试模式下编译。在开发模式下，Express 视图会在每个请求中进行编译，而在生产环境中，它们会被缓存。


**此环境变量是外部库中广泛使用的约定，但 Node.js 本身并未使用**。


# Node.js 与 WebAssembly


[**WebAssembly**](https://webassembly.org/)是一种高性能的类汇编语言，可以从多种语言进行编译，包括 C/C++、Rust 和 AssemblyScript。目前，Chrome、Firefox、Safari、Edge 和 Node.js 支持它！


WebAssembly 规范详细介绍了两种文件格式，一种是称为 WebAssembly 模块（扩展名为**.wasm**的二进制格式，另一种是相应的文本表示形式（称为 WebAssembly 文本格式），扩展名为**.wat** 。


## 关键概念

- 模块 - 已编译的 WebAssembly 二进制文件，即**.wasm**文件。
- 内存 - 可调整大小的 ArrayBuffer。
- 表 - 未存储在内存中的可调整大小的类型化引用数组。
- 实例 - 模块及其内存、表和变量的实例化。

为了使用 WebAssembly，您需要一个**.wasm**二进制文件和一组 API 来与 WebAssembly 进行通信。 Node.js 通过全局**WebAssembly**对象提供必要的 API。


```javascript
console.log(WebAssembly);
/*
Object [WebAssembly] {
  compile: [Function: compile],
  validate: [Function: validate],
  instantiate: [Function: instantiate]
}
*/
```


## 生成 WebAssembly 模块


有多种方法可用于生成 WebAssembly 二进制文件，包括：

- 手动编写 WebAssembly ( **.wat** ) 并使用[wabt](https://github.com/webassembly/wabt)等工具转换为二进制格式
- 将[emscripten](https://emscripten.org/)与 C/C++ 应用程序结合使用
- 将[wasm-pack](https://rustwasm.github.io/wasm-pack/book/)与 Rust 应用程序结合使用
- 如果您喜欢类似 TypeScript 的体验，请使用[AssemblyScript](https://www.assemblyscript.org/)

## 如何使用


一旦有了 WebAssembly 模块，就可以使用 Node.js **WebAssembly**对象来实例化它。


```javascript
// Assume add.wasm file exists that contains a single function adding 2 provided arguments
const fs = require('node:fs');

const wasmBuffer = fs.readFileSync('/path/to/add.wasm');
WebAssembly.instantiate(wasmBuffer).then(wasmModule => {
  // Exported function live under instance.exports
  const { add } = wasmModule.instance.exports;
  const sum = add(5, 6);
  console.log(sum); // Outputs: 11
});
```


# 调试 Node.js


## 启用检查器


当使用**--inspect**开关启动时，Node.js 进程会侦听调试客户端。默认情况下，它将侦听主机和端口 127.0.0.1:9229。每个进程还被分配了一个唯一的[UUID](https://tools.ietf.org/html/rfc4122) 。


Inspector 客户端必须知道并指定要连接的主机地址、端口和 UUID。完整的 URL 看起来像 **ws://127.0.0.1:9229/0f2c936f-b1cd-4ac9-aab3-f63b0f33d55e** 。


如果 Node.js 收到**SIGUSR1**信号，它也会开始侦听调试消息。 （ **SIGUSR1**在 Windows 上不可用。）在 Node.js 7 及更早版本中，这会激活旧版调试器 API。在 Node.js 8 及更高版本中，它将激活 Inspector API。


## 安全影响


由于调试器可以完全访问 Node.js 执行环境，因此能够连接到此端口的恶意攻击者可能能够代表 Node.js 进程执行任意代码。


## 公开暴露调试端口是不安全的


如果调试器绑定到公共 IP 地址或 0.0.0.0，则任何可以访问您的 IP 地址的客户端都将能够不受任何限制地连接到调试器，并且能够运行任意代码。


默认情况下， **node --inspect**绑定到 127.0.0.1。如果您打算允许与调试器的外部连接，则明确需要提供公共 IP 地址或 0.0.0.0 等。这样做可能会让您面临潜在的重大安全威胁。我们建议您确保适当的防火墙和访问控制到位，以防止安全漏洞。


## 本地应用程序可以完全访问检查器


即使您将检查器端口绑定到 127.0.0.1（默认值），在您的计算机上本地运行的任何应用程序都将具有不受限制的访问权限。这是设计使然，以便本地调试器能够方便地附加。


## 浏览器、WebSocket 和同源策略


在网络浏览器中打开的网站可以在浏览器安全模型下发出 WebSocket 和 HTTP 请求。需要初始 HTTP 连接才能获取唯一的调试器会话 ID。同源策略阻止网站建立此 HTTP 连接。为了针对[DNS 重新绑定攻击](https://en.wikipedia.org/wiki/DNS_rebinding)提供额外的安全性，Node.js 会验证连接的“主机”标头是否精确指定 IP 地址或**localhost** 。


这些安全策略不允许通过指定主机名连接到远程调试服务器。您可以通过指定 IP 地址或使用 ssh 隧道来解决此限制。


## Error 处理

错误处理是一种尽快发现和解决错误的方法。Node.js 中的错误可以分为操作错误或程序员错误。


# Node.js 中的错误处理是什么


错误到底是什么？它是一种查看代码中漏洞的方式。遵循这种逻辑，错误处理是一种找到这些漏洞并尽快解决它们的方法。


从这个解释可以明显看出，难点在于为错误处理建立一个良好的基础。归根结底，这一切都是为了让你在工作结束时保持理智。正确处理错误意味着开发一个健壮的代码库，并通过轻松发现漏洞和错误来减少开发时间。


# 错误类型：操作错误与程序员错误


## 操作错误


操作错误代表运行时问题。这些错误在Node.js运行时是预期的，应该以适当的方式处理。这并不意味着应用程序本身有漏洞。只是表示这些错误需要被正确处理。以下是一些常见的操作错误列表：

- 无法连接到服务器
- 无法解析主机名
- 用户输入无效
- 请求超时
- 服务器返回 500 响应
- 套接字挂起
- 系统内存不足

## 程序员错误


程序员错误就是我们所说的漏洞。它们代表了代码本身的问题。以下是一个Node.js中常见的错误，当你尝试读取未定义对象的属性时。这是一个经典的程序员错误案例。还有一些其他的例子：

- 调用不带回调的异步函数
- 未完成一个Promise
- 没有捕捉到被拒绝的Promise
- 在需要对象的地方传递了一个字符串
- 传递了一个需要字符串的对象
- 在函数中传递了错误的参数

# 什么是错误对象？


错误对象是 Node.js 运行时中的内置对象。当错误发生时，它会为您提供一组有关错误的信息。


基本错误如下所示：


```javascript
const error = new Error("An error message")
console.log(error.stack)
```


它有一个 error.stack 字段，可以为你提供一个堆栈追踪，显示错误的来源。它还列出了在错误发生之前调用的所有函数。error.stack 字段在调试时非常有用，因为它也会打印出 error.message。


# 如何处理 Node.js 中的错误：您应该遵循的最佳实践


在 Promise 中使用 async/await 和 try-catch 语句，或 .catch() 错误。

1. 使用自定义错误处理操作错误

    使用 async/await 模式可以编写看起来像同步代码，但实际上是异步的。


    ```javascript
    const anAsyncTask = async () => {
     try {
     	const user = await getUser()
     	const cart = await getCart(user)
    
     	return cart
     } catch (error) {
     	console.error(error)
     } finally {
     	await cleanUp()
     }
    }
    ```


    这种模式将清理你的代码并避免可怕的回调地狱。


    可以使用 Node.js 中的内置 Error 对象，因为它为您提供有关堆栈跟踪的详细信息。


    这是一个名为 baseError.js 的文件，您可以在其中为将使用的每个自定义错误设置基数。


    ```javascript
    // baseError.js
    
    class BaseError extends Error {
     constructor (name, statusCode, isOperational, description) {
     super(description)
    
     Object.setPrototypeOf(this, new.target.prototype)
     this.name = name
     this.statusCode = statusCode
     this.isOperational = isOperational
     Error.captureStackTrace(this)
     }
    }
    
    module.exports = BaseError
    ```


    还要创建一个 httpStatusCodes.js 文件来保存要处理的所有状态代码的映射。


    ```javascript
    // httpStatusCodes.js
    
    const httpStatusCodes = {
     OK: 200,
     BAD_REQUEST: 400,
     NOT_FOUND: 404,
     INTERNAL_SERVER: 500
    }
    
    module.exports = httpStatusCodes
    ```


    然后，可以创建一个 api404Error.js 文件，并使用自定义错误扩展 BaseError 以处理 404。


    ```javascript
    // api404Error.js
    
    const httpStatusCodes = require('./httpStatusCodes')
    const BaseError = require('./baseError')
    
    class Api404Error extends BaseError {
     constructor (
     name,
     statusCode = httpStatusCodes.NOT_FOUND,
     description = 'Not found.',
     isOperational = true
     ) {
     super(name, statusCode, isOperational, description)
     }
    }
    
    module.exports = Api404Error
    ```


    当您想要处理 404 错误时，将其放入代码中。


    ```javascript
    const Api404Error = require('./api404Error')
    
    ...
    const user = await User.getUserById(req.params.id)
    if (user === null) {
     throw new Api404Error(`User with id: ${req.params.id} not found.`)
    }
    ...
    ```


    您可以为任何自定义错误、500、400 以及您想要处理的任何其他错误复制此代码。

2. 使用中间件

    一旦拥有了一组自定义错误，就可以配置集中式错误处理。你需要有一个中间件来捕获所有错误。在那里你可以决定如何处理这些错误，以及如果需要通过警报通知你，该将这些错误发送到哪里。


    在你的 API 路由中，你将最终使用 next() 函数将错误转发到错误处理中间件。


    ```javascript
    ...
    app.post('/user', async (req, res, next) => {
     try {
    const newUser = User.create(req.body)
     } catch (error) {
     next(error)
     }
    })
    ...
    ```


    `next()` 函数是 Express.js 中间件中的一个特殊函数，用于将值传递到中间件链的下一个环节。在路由文件的底部，你应该有一个 `use()` 方法，该方法使用错误处理中间件函数。


    ```javascript
    const { logError, returnError } = require('./errorHandler')
    
    app.use(logError)
    app.use(returnError)
    ```


    错误处理中间件应包括几个关键部分。你应该检查错误是否是操作错误，并决定哪些错误需要发送为警报通知，以便你可以更详细地进行调试。以下是我建议你添加到错误处理程序中的内容。


    ```javascript
    function logError (err) {
     console.error(err)
    }
    
    function logErrorMiddleware (err, req, res, next) {
     logError(err)
     next(err)
    }
    
    function returnError (err, req, res, next) {
     res.status(err.statusCode || 500).send(err.message)
    }
    
    function isOperationalError(error) {
     if (error instanceof BaseError) {
     return error.isOperational
     }
     return false
    }
    
    module.exports = {
     logError,
     logErrorMiddleware,
     returnError,
     isOperationalError
    }
    ```

3. 优雅地重启您的应用程序以处理程序员错误

    这些错误往往会导致应用程序出现问题，如内存泄漏和高CPU使用率。最好的解决方法是使应用程序崩溃并通过使用Node.js集群模式或像PM2这样的工具优雅地重新启动。

4. 捕获所有未捕获的异常

    当发生此类意外错误时，您希望通过发送通知并重新启动应用程序来立即处理，以避免意外行为。


    ```javascript
    const { logError, isOperationalError } = require('./errorHandler')
    
    ...
    process.on('uncaughtException', error => {
     logError(error)
    
     if (!isOperationalError(error)) {
     process.exit(1)
     }
    })
    ...
    ```

5. 捕获所有未处理的 Promise 拒绝

    Node.js 中的 Promise 拒绝只会导致警告。您希望它们抛出错误，以便您可以正确处理它们。


    最好使用后备方案并订阅：


    ```javascript
    process.on('unhandledRejection', callback)
    ```


    这可以让你正确地抛出错误。


    错误处理流程应如下所示。


    ```javascript
    ...
    const user = User.getUserById(req.params.id)
     .then(user => user)
     // missing a .catch() block
    ...
    
    // if the Promise is rejected this will catch it
    process.on('unhandledRejection', error => {
     throw error
    })
    
    process.on('uncaughtException', error => {
     logError(error)
    
     if (!isOperationalError(error)) {
     process.exit(1)
     }
    })
    ```

6. 使用集中位置存储日志和错误警报

    使用结构化日志记录以格式化的方式打印错误，并将它们发送到中央位置进行保存，比如我们的日志管理工具Sematext Logs。


    要做到这一点，你应该使用像 winston 和 morgan 这样的日志记录器。此外，你还可以添加 winston-logsene，以便立即将日志发送到 Sematext。


    首先，为 winston 和 winston-logsene 创建一个设置。创建一个 loggers 目录和一个 logger.js 文件。将以下内容粘贴到该文件中。


    ```javascript
    // logger.js
    
    const winston = require('winston')
    const Logsene = require('winston-logsene')
    
    const options = {
     console: {
     level: 'debug',
     handleExceptions: true,
     json: false,
     colorize: true
     },
     logsene: {
     token: process.env.LOGS_TOKEN,
     level: 'debug',
     type: 'app_logs',
     url: 'https://logsene-receiver.sematext.com/_bulk'
     }
    }
    
    const logger = winston.createLogger({
     levels: winston.config.npm.levels,
     transports: [
     new winston.transports.Console(options.console),
     new Logsene(options.logsene)
     ],
     exitOnError: false
    })
    
    module.exports = logger
    ```


    这样做的好处是您可以获得 JSON 格式的日志，您可以对其进行分析以获得有关应用程序的更多有用信息。您还可以将所有日志转发到 Sematext。每当发生错误时，这都会提醒您。


    此外，您应该在loggers目录中添加httpLogger.js文件，并添加[morgan](https://github.com/expressjs/morgan)和[morgan-json](https://github.com/indexzero/morgan-json)来打印访问日志。将其粘贴到 httpLogger.js 中：


    ```javascript
    const morgan = require('morgan')
    const json = require('morgan-json')
    const format = json({
     method: ':method',
     url: ':url',
     status: ':status',
     contentLength: ':res[content-length]',
     responseTime: ':response-time'
    })
    
    const logger = require('./logger')
    const httpLogger = morgan(format, {
     stream: {
     write: (message) => {
     const {
     method,
     url,
     status,
     contentLength,
     responseTime
     } = JSON.parse(message)

     timestamp: new Date().toString(),

     url,
     status: Number(status),
     contentLength,
     responseTime: Number(responseTime)
     })
     }
     }
    })
    
    module.exports = httpLogger
    ```


    在您的 app.js 文件中，您现在可以同时需要 logger.js 和 httpLogger.js，并使用记录器而不是 console.log()。


    ```javascript
    // app.js
    
    const logger = require('./loggers/logger')
    const httpLogger = require('./loggers/httpLogger')
    ...
    
    app.use(httpLogger)
    ...
    ```


    在 errorHandler.js 中，您现在可以将所有 console.error() 语句替换为 logger.error()，以将日志保留在 Sematext 中。


    ```javascript
    // errorHandler.js
    
    const logger = require('../loggers/logger')
    const BaseError = require('./baseError')
    
    function logError (err) {
     logger.error(err)
    }
    
    function logErrorMiddleware (err, req, res, next) {
     logError(err)
     next(err)
    }
    
    function returnError (err, req, res, next) {
     res.status(err.statusCode || 500).send(err.message)
    }
    
    function isOperationalError(error) {
     if (error instanceof BaseError) {
     return error.isOperational
     }
     return false
    }
    
    module.exports = {
     logError,
     logErrorMiddleware,
     returnError,
     isOperationalError
    }
    ```


# 如何传递错误：函数模式


Node.js 中传递错误的四种主要方法：

- 抛出错误（使其成为异常）
- 将错误传递给回调，这是专门为处理错误和异步操作结果而提供的函数
- 将错误传递给拒绝 Promise 函数
- 在 EventEmitter 上发出“错误”事件

## 抛出错误


当您抛出错误时，它会展开整个函数调用堆栈，忽略堆栈上的任何函数。它在调用函数的同一上下文中同步传递。


如果您使用 try-catch 块，您可以优雅地处理错误。否则，应用程序通常会崩溃，除非您有捕获未捕获异常的后备措施。


下面是抛出错误并在 try-catch 块中处理错误的示例：


```javascript
const getUserWithAsyncAwait = async (id) => {
 try {
 const user = await getUser(id)
 if (!user) {
 throw new 404ApiError('No user found.')
 }

 return user
 } catch (error) {
 // handle the error
 logError(error)
 }
}

const user = await getUserWithAsyncAwait(1)

...
```


## 使用回调


回调是异步传递错误的最基本方式。用户向您传递一个函数 - 回调，您可以在异步操作完成后的某个时间调用它。通常的模式是回调被调用为callback(err, result)，其中只有err和result之一是非空的，具体取决于操作是成功还是失败。


您将回调函数作为参数传递给调用函数，稍后在异步函数完成执行时调用该函数。


通常的模式如下所示：


```javascript
callback(err, result)
```


回调中的第一个参数始终是错误。


在回调函数内部，您将首先检查错误是否存在，只有当它是非空值时，您才继续执行回调函数。


```javascript
function getUserWithCallback(id, callback) {
 getUser(id, function(user) {
 if (!user) {
 return callback(new 404ApiError('No user found.'))
 }

 callback(null, user)
 })
}

getUserWithCallback(1, function(err, user) {
 if (err) {
 // handle the error
 logError(error)
 }

 const user = user
 ...
})
```


## 使用 Promise


Promise 已经取代了回调，成为编写异步代码的新的和改进的方式。


自 Node.js 版本 8 包含开箱即用的 async/await 以来，这种模式已成为新规范。异步代码可以写得像同步代码。可以使用 try-catch 来捕获错误。


```javascript
function getUserWithPromise(id) {
 return new Promise((resolve, reject) => {
 getUser(id, function(user) {
 if (!user) {
 return reject(new 404ApiError('No user found.'))
 }

 resolve(user)
 })
 })
}

getUserWithPromise(1)
 .then(user => {
 const user = user
 ...
 })
 .catch(err => {
 logError(error)
 })
```


## 使用 EventEmitter


在某些情况下，你不能依赖于promise拒绝或回调。那么当你从流中读取文件时该怎么办呢？或者从数据库中获取行并在它们到达时读取它们呢？我每天都会遇到的一个用例是流式传输日志行并在它们到达时进行处理。

## Node.js 模块

我们将代码分割到不同的文件中，以便维护、组织和重用代码。模块系统使我们能够分割和包含代码，并在需要时导入其他开发者编写的代码。简单来说，模块不过是一个 JavaScript 文件。Node.js 有许多内置模块，它们是平台的一部分，随 Node.js 安装一起提供，例如 HTTP、fs、path 等等。


CommonJS 和 ES（ECMAScript）是 Node 中使用的模块系统。CommonJS 是默认的模块系统。然而，最近在 NodeJS 中添加了一个新的模块系统——ES 模块。CommonJS 模块使用 require() 语句来导入模块，使用 module.exports 来导出模块，而 ES 模块则使用 import 和 export 来进行模块的导入和导出。


# Node.js 中的 CommonJS 与 ES 模块


在现代软件开发中，模块将代码组织成独立的块，从而构成更大、更复杂的应用程序。在浏览器JavaScript生态系统中，JavaScript模块的使用依赖于import和export语句；这些语句分别加载和导出 ECMAScript 模块（或 ES 模块）。


ES 模块格式是打包 JavaScript 代码以供重用的官方标准格式，大多数现代 Web 浏览器本身都支持该模块。


然而，Node.js 默认支持 CommonJS 模块格式。 CommonJS 模块使用require加载，变量和函数使用module.exports。


随着 JavaScript 模块系统的标准化，ES 模块格式在 Node.js v8.5.0 中引入。作为实验性模块，需要-- experimental - modules才能在 Node.js 环境中成功运行 ES 模块。不过，Node.js 从 13.2.0 版本开始就稳定支持 ES 模块。


CommonJS 主要用于使用 Node.js 进行服务器端开发。它使用require和module.exports 。另一方面，异步模块定义 (AMD) 专注于使用define和require进行异步加载的浏览器环境，这提高了页面加载时间和响应能力。尽管如此，仍然需要更好的解决方案。对可在服务器端和浏览器环境中运行的解决方案的需求促使了通用模块定义 (UMD) 的开发。


然后是 ES 模块，它为客户端和服务器端 JavaScript 提供本机模块系统。 ES6 模块提供了清晰的语法、导入和导出语句，并支持异步加载。这一进步使代码更易于维护、可重用和高性能，从而使开发人员能够构建更具可扩展性的应用程序。


# CommonJS vs ES 模块的语法


默认情况下，Node.js 将 JavaScript 代码视为 CommonJS 模块。因此，CommonJS 模块的特点是使用 require 语句来导入模块，并使用 module.exports 来导出模块。


例如，这是一个导出两个函数的 CommonJS 模块：


```javascript
module.exports.add = function(a, b) {
        return a + b;
} 

module.exports.subtract = function(a, b) {
        return a - b;
}
```


**我们还可以使用require将公共函数导入到另一个 Node.js 脚本中，就像我们在这里所做的那样：**


```javascript
const {add, subtract} = require('./util')

console.log(add(5, 5)) // 10
console.log(subtract(10, 5)) // 5
```


另一方面，库的作者也可以通过将文件扩展名从 .js 更改为 .mjs 来简单地在 Node.js 包中启用 ES 模块。例如，以下是一个简单的 ES 模块（具有 .mjs 扩展名），它导出了两个供公众使用的函数：


```javascript
// util.mjs

export function add(a, b) {
        return a + b;
}

export function subtract(a, b) {
        return a - b;
}
```


然后我们可以使用import语句导入这两个函数：


```javascript
// app.mjs

import {add, subtract} from './util.mjs'

console.log(add(5, 5)) // 10
console.log(subtract(10, 5)) // 5
```


另一种在项目中启用 ES 模块的方法是在最近的 package.json 文件（与您正在制作的软件包位于同一文件夹）中添加一个 "type: module" 字段：


```javascript
{
  "name": "my-library",
  "version": "1.0.0",
  "type": "module",
  // ...
}
```


通过这种包含，Node.js 会将该包中的所有文件视为 ES 模块，并且您不必将文件更改为 .mjs 扩展名。


另外，可以安装并设置像 Babel 这样的转译器，将您的 ES 模块语法编译为 CommonJS 语法。像 React 和 Vue 这样的项目支持 ES 模块，因为它们在底层使用 Babel 来编译代码。


# 在 Node.js 中使用 ES 模块和 CommonJS 模块的优缺点


ES 模块格式的创建是为了标准化 JavaScript 模块系统。它已成为封装 JavaScript 代码以供重用的标准格式。、


另一方面，CommonJS 模块系统内置于 Node.js 中。在 Node.js 中引入 ES 模块之前，CommonJS 是 Node.js 模块的标准。因此，出现了许多使用 CommonJS 编写的 Node.js 库和模块。


对于浏览器支持，所有主流浏览器都支持 ES 模块语法，您可以在 React 和 Vue.js 等框架中使用import / export 。这些框架使用像 Babel 这样的转译器来将import / export语法编译为require ，旧版 Node.js 版本本身就支持 require 。


除了成为 JavaScript 模块的标准之外，ES 模块语法也比require更具可读性。由于语法相同，主要在客户端编写 JavaScript 的 Web 开发人员在使用 Node.js 模块时不会出现任何问题。


# CommonJS 提供模块导入的灵活性


在 ES 模块中，只能在文件开头调用 import 语句。在其他任何地方调用它会自动将表达式移动到文件开头，甚至可能引发错误。另一方面， require函数在运行时被解析。因此， require可以在代码中的任何位置调用。您可以使用它从if语句、条件循环和函数有条件或动态加载模块。


# CommonJS 同步加载模块，ES 模块异步


使用require的局限性之一是它同步加载模块，这意味着模块是被一一加载和处理的。只有前一个任务完成后，一项任务才能开始。这称为“阻塞”，因为如果某个操作需要很长时间才能完成，则会阻止后续任务启动。


同步代码编写和读取都很简单，并且遵循线程执行模型，可以更轻松地预测代码流程和结果。然而，它可能会导致严重的性能问题，因为同步加载可能会导致整个应用程序或程序冻结或变得无响应，特别是在涉及 I/O 操作、长时间计算或实时响应性的场景中，导致用户体验和可扩展性较差。


在这种情况下，由于import异步行为，其性能可能会优于require 。然而，对于使用几个模块的小型应用程序来说， require的同步特性可能不是什么大问题。


另一方面，异步操作通常使用回调、promise 或 async/await 语法来执行。与同步代码不同，异步代码可能更难读取、编写和调试，因为它是非线性的。但它提供了更好的用户体验，因为它在高流量 Web 应用程序上运行良好，无需等待轮到运行。


# CommonJS 模块


在模块中，当您定义变量、函数和类时，这些标识符只能在该模块中访问。


要在另一个模块中使用这些对象，您需要使用以下导出机制之一导出它们或将它们组合起来：

- 命名导出
- 默认导出

## 命名导出


在命名导出中，您可以将标识符分配给module.exports对象的各个属性。这允许您从一个模块导出多个标识符并将它们单独导入到另一个模块中。


首先，创建一个名为math.js的新模块，其中包含两个函数add和subtract ：


```javascript
function add(a, b) {
  return a + b;
}

function subtract(a, b) {
  return a - b;
}
```


这 add 和 subtract 函数仅在 math.js 模块。


要在另一个模块中使用这些函数，您需要通过将它们分配给module.exports对象的属性来导出它们。


二、导出 add 和 subtract 函数从 math.js 模块：


```javascript
// math.js
function add(a, b) {
  return a + b;
}

function subtract(a, b) {
  return a - b;
}

module.exports.add = add;
module.exports.subtract = subtract;
```


在分配中，您使用不带括号的函数名称，并将它们分配给module.exports对象的各个属性。


请注意，导出函数时可以使用不同的名称，如下所示：


```javascript
module.exports.sum = add;
module.exports.deduct = subtract;
```


目前，我们在导出时将保留相同的函数名称。


第三，创建一个名为app.js的新模块，并将math模块导入到app.js模块中：


```javascript
const math = require('./math');

console.log(math);
```


要导入模块，请使用require()函数。 require()函数接受不带或不带.js扩展名的模块路径。


./表示math.js模块与app.js模块位于同一目录中。


控制台显示math是一个对象，包含两个函数add和subtract作为其属性：


```javascript
{ add: [Function: add], subtract: [Function: subtract] }
```


最后，通过 math 对象调用加和减函数：


```javascript
// app.js
const math = require('./math');

let a = 20,
    b = 5;

let result = math.add(a, b);
console.log(`${a} + ${b} = ${result}`);

result = math.subtract(a, b);
console.log(`${a} - ${b} = ${result}`);
```


为了使代码更加简洁，您可以在app.js模块中使用对象解构，如下所示：


```javascript
const { add, subtract } = require('./math');
```


在此语法中，我们将require()函数返回的对象的属性解构为add和subtract变量。


它相当于以下内容：


```javascript
// app.js

const math = require('./math');

const add = math.add,
      subtract = math.subtract;

// ...
```


## 默认导出


默认导出允许您从模块导出单个标识符（变量、函数、对象或类）。一个模块只能有一个默认导出。


下面显示了如何在 math.js 模块中使用默认导出：


```javascript
// math.js

function add(a, b) {
  return a + b;
}

function subtract(a, b) {
  return a - b;
}

module.exports = add;
```


在此示例中，我们将add函数分配给module.exports对象而不是其属性。这称为默认导出。


在app.js模块中，您可以直接从math.js文件中导入add函数：


```javascript
// app.js 
const add = require('./math');

let a = 20,
  b = 5;

let result = add(a, b);
console.log(`${a} + ${b} = ${result}`);
```


## 混合默认导出和命名导出


首先，使用默认导出和命名导出在math.js模块中导出标识符：


```javascript
// math.js

function add(a, b) {
  return a + b;
}

function subtract(a, b) {
  return a - b;
}

module.exports = add;
module.exports.subtract = subtract;
```


其次，将 math.js 中的加法和减法函数导入到 app.js 模块中：


```javascript
const math = require('./math');

const add = math;
const subtract = math.subtract;
// ...
```


## 了解模块包装函数


在执行模块之前，Node.js 使用函数包装器包装该模块内的所有代码，如下所示：


```javascript
(function(exports, require, module, __filename, __dirname) {
    // Module code
});
```


例如，执行前math.js模块的代码如下所示：


```javascript
(function (exports, require, module, __filename, __dirname) {
   function add(a, b) {
  return a + b;
}

function subtract(a, b) {
  return a - b;
}

module.exports.add = add;
module.exports.subtract = subtract;

});
```


通过这样做，Node.js 实现了以下重要目标：

- 保持顶级变量（ var 、 let和const ）的作用域为模块而不是全局对象。
- 创建一些特定于模块的变量，例如全局变量，例如module和exports 。

请注意， exports 对象引用 module.exports :


```javascript
console.log(module.exports === exports); // true
```


## 多次导入同一个模块


当您使用require()函数多次导入同一模块时， require()函数仅在第一次调用时对模块求值一次，并将其放入缓存中。


在后续调用中， require()函数使用缓存中的exports对象，而不是再次执行该模块。


以下示例说明了其工作原理：


首先，创建一个名为dblogger.js的新模块，其中包含以下代码：


```javascript
console.log('Connected to the DB');
```


其次，在app.js中导入dblogger.js模块两次：


```javascript
let dbLogger = require('./dblogger');
dbLogger = require('./dblogger');
```


Output: 输出：


```javascript
DBLogger is loaded.
```


在此示例中，您可以看到消息'DBLogger is loaded.'只有一次，没有两次。这意味着 Node.js 仅评估dblogger.js一次。


## 总结

- 在 CommonJS 模块中，Node.js 将 JavaScript 文件视为模块。
- 在模块中声明的所有标识符（包括变量、常量、函数和类）的作用域都是该模块，而不是全局作用域。
- 通过将标识符分配给module.exports对象的属性来从模块导出标识符。
- Node.js 在执行之前将模块代码包装在模块包装函数中。
- Node.js 仅执行一个模块一次，并将结果放入缓存中以供下次使用。

# ES 模块


对于ES模块，模块是使用import和export关键字定义的，而不是CommonJS中的require ()函数。以下是它们的使用方法：


```javascript
export function sayLanguage(language) {
    console.log(`I love ${language}!`);
  }
```

## ES 模块中的导入和导出语法


在 ES 模块中，说明符就像在from关键字之后使用的基于字符串的文件路径。有两种算法可以加载 ES 模块说明符并确定已解析 URL 的模块格式。示例如下所示：


```javascript
import {cat} from 'animals';
```


例如，本例中的animal说明符是一个 npm 包。可以引用说明符的其他方式包括从绝对和相对文件路径或 URL 以及其他包内的路径。

> 注意：从版本 14.13.1 开始，Node.js 内置模块现在可以通过绝对 URL 字符串加载或引用。使用 import 关键字时，必须提供文件扩展名来解析相对或绝对 URL 说明符。

尽管import语句仅允许在 ES 模块中使用，但它们可以引用 ESM 或 CommonJS 模块。例如：


```javascript
import packageMain from 'commonjs-package'; // Works

import { method } from 'commonjs-package'; // Errors
```


对于 ES 模块中的导出，我们可以使用以下内容：

- 命名导出

    ```javascript
    module.exports.name = "Alex"
    ```

- 默认导出

    ```javascript
    export default function sayName() {console.log('My name is Mat')}
    ```

> 注意：所有内置 Node 包都支持上述所有类型的导出（命名、默认）。从 Node 版本 14.13.0 开始，添加了对 ESM 中 CommonJS 命名导出检测的支持。

## 包入口点


现在在项目的 package.json 文件中有两个可以定义入口点的字段：main 和 exports。main 字段仅定义包的主要入口点，而 exports 字段则提供了一种替代方法，它不仅可以定义主要入口点，还可以提供封装包的好处。


可以通过在包名称后附加路径来访问包中的模块文件。另一种方式是，如果包的 package.json 文件中包含一个导出字段，则只能通过导出对象中定义的路径来访问包中的文件。


为了设置包的主要入口点，例如，建议在包的 package.json 文件中同时定义 exports 和 main。


# 两种模块系统的特点

- CommonJS 和 ES 模块都支持动态import () 。它可用于包含 CommonJS 代码中的 ES 模块文件
- ECMAScript 6 还规定可以从 URL 加载模块，而 CommonJS 仅限于相对和绝对文件路径。
- Node.js 无法理解的格式的源代码可以转换为 JavaScript。
- 动态导入可用于导入 ES 和 CommonJS 模块。在 CommonJS 模块中，它可用于加载 ES 模块。请注意，它返回一个promise
- 使用import关键字时必须提供文件扩展名。必须完全指定目录索引（例如'./database/index.js' ）
- 现在可以通过使用[条件导出来实现 CommonJS 和 ESM 的双重使用。](https://nodejs.org/api/esm.html#esm_conditional_exports)现在，Node.js 可以运行 ES 模块入口点，并且包可以同时包含 CommonJS 和 ESM 入口点
- 从版本 14.13.1 开始，ESM 添加了对使用node : URLs加载 Node.js 内置模块的node : URLs ，允许通过有效的绝对 URL 字符串引用内置模块
