# 什么是设计模式

设计模式是软件设计中常见问题的可重复使用的解决方案。它们代表了经验丰富的面向对象软件开发人员使用的最佳实践。这些模式可以通过提供解决常见问题的行之有效的方法来加快开发过程。

# 顶级设计模式

## **1. 单例模式**

单例模式是一种设计模式，它将类的创建限制为**仅一个实例**。这在需要单点控制或协调的场景中很有用。换句话说，它确保一个类只有一个实例，并提供对它的全局访问点。

单例模式的一个实际用途是管理大型应用（如 Web 应用）中的配置设置对象。这可确保仅存在一个配置对象实例（保存数据库字符串和 API 密钥等设置），从而提供单一访问点并防止发生冲突。

此模式通常用于配置数据、缓存、连接池或日志记录，在这些情况下，运行一个实例供应用程序中的其他进程使用会更有效。当需要维护状态、初始化字段或管理调用和回调队列时，它也很有用。

例如，如果应用程序有一个可从各个地方访问的下拉列表，则 Singleton 可以管理此共享资源。这确保如果在一个地方修改了列表，则更改会反映在整个应用程序中。

**示例：**

下面是用 JavaScript 实现下拉列表的一个基本示例：

```javascript
let dropdownListItems;
class DropdownList {
    constructor(items = []) {
        if (dropdownListItems) {
            return dropdownListItems;
        }

        dropdownListItems = items;
    }

    addItem(item) {
        dropdownListItems.push(item);
    }

    removeItem(item) {
        dropdownListItems = dropdownListItems.filter(i => i !== item);
    }
}

const dropdownList = new DropdownList();
export default dropdownList;
```

这段 JavaScript 代码定义了一个类 `DropdownList` 及其一个实例。

1. `dropdownListItems`是一个最初为undefined 的变量。
2. 该类 `DropdownList`定义有一个构造函数，该构造函数接受一个项目数组作为参数，如果没有提供参数，则默认为空数组。
3. 在构造函数中，如果 `dropdownListItems`已经定义，则返回已定义的。如果`dropdownListItems`未定义，则将`items`参数分配给`dropdownListItems`。
4. 此类 `DropdownList`有两种方法：`addItem`和`removeItem`。
    - `addItem`方法：此方法接受一个项目作为参数并将其推送到数组`dropdownListItems`。
    - `removeItem`方法：此方法接受一个项目作为参数并将其从数组中删除`dropdownListItems`。它通过重新分配`dropdownListItems`到新数组来过滤要删除的项目。
5. `DropdownList`的实例在创建时没有参数，因此将用一个空数组初始化。此实例存储在常量中`dropdownList`。

## 2.外观模式

外观模式是一种结构设计模式，它为更复杂的底层系统、库或框架提供简化的接口。它有助于抽象复杂性并为客户端提供更易于使用和理解的界面。

考虑这样一种情况：你的代码需要与涉及大量对象的复杂库或框架进行交互。通常，必须初始化这些对象、管理依赖项并确保方法按正确的顺序执行。此过程可能会导致业务逻辑与第三方类的细节紧密交织在一起，从而使代码难以理解和维护。

外观是复杂子系统的简化接口，仅提供必要的功能。在使用只需要一小部分功能的复杂库时，外观非常有用。

### 真实案例场景及实施

以一个音乐播放器应用为例。在后台，可能会发生一系列复杂的操作，例如加载媒体文件、解码音频、管理音频缓冲区以及将音频流式传输到设备的输出。但是，从用户的角度来看，他们只与一个简单的界面进行交互 - 播放、暂停或停止。

在这种情况下，可以使用 JavaScript 实现外观，如下所示：

```javascript
class MusicPlayer {
    constructor() {
        this.audioContext = new AudioContext();
        this.audioBuffer = null;
        // other complex initializations...
    }play() {
        // handle complex operations
    }pause() {
        // handle complex operations
    }stop() {
        // handle complex operations
    }
}
```

有了这个外观，客户端代码可以简单地创建一个新`MusicPlayer`实例并调用`play()`、`pause()`或`stop()`，而不必担心底层的复杂性。

## 3. 观察者模式

观察者模式是一种易于理解且广泛使用的消息传递设计，其中对象（称为主体）维护其依赖项（称为观察者）的列表，并自动通知它们任何状态变化。这促进了相关对象之间的松散耦合，并允许它们之间进行有效的通信。

### 真实案例

假设有一位顾客和一家商店。顾客想要一款商店里还没有的新款 iPhone。他们可以每天查看，但大多数行程都是不必要的。或者，商店可以在新产品到货时向所有顾客发送垃圾邮件，这样他们就不必无谓地跑一趟，但会惹恼其他人。这就造成了一个两难境地：要么顾客浪费时间，要么商店浪费资源。

### 观察者模式解决方案

在上面的例子中，客户是观察者，而商店是主体。客户可能希望接收特定类别产品的更新，也可能不希望接收，因此客户可以选择要观察的内容。

在这种情况下，商店（即主体）知道谁对其更新感兴趣（客户或观察者），并且可以有许多客户。各种客户都可以从同一个商店注册更新，并且每当添加新产品时，所有客户都会收到通知。

每次在商店中创建、更新或删除新产品时都会发生这种沟通。

尽管在观察者模式中 Store 知道其客户是谁，但它并不关心或知道每个客户将如何处理收到的更新。每个客户都可以完全控制如何处理他们收到的更新。

### 代码示例

```javascript
// Define a class for the Provider
class Store {
  constructor() {
    this.observers = [];
  }// Add an observer to the list
  add(observer) {
    this.observers.push(observer);
  }// Notify all observers about new product
  notify(product) {
    this.observers.forEach(observer => observer.update(product));
  }
}// Define a class for the Observer
class Customer {
  update(product) {
    console.log(`New product added: ${product}`);
  }
}// Usage
const store = new Store();
const customer = new Customer();
store.add(customer); // Add a customer to the store's observers
store.notify('iPhone 15'); // Notify all observers about a new product
```

在上面的代码中，通过创建两个类`Store`和来举例说明观察者模式`Customer`。

该类`Store`代表提供者。它有一个数组，用于存储对更新感兴趣的`observers`所有观察者（实例） 。它还有一个添加新观察者的方法和一个通知所有观察者有关新产品的方法。

虽然该类`Customer`代表观察者，但它有一个`update`方法，当它收到更新时会执行一些操作，在这种情况下，它会登录控制台。

一切就绪，让我们使用它。因此，我们创建一个`Store`和一个`Customer`对象。使用方法`add`将`Customer`添加到`Store`的观察者中。

然后，使用 该方法`Store`使用`notify`通知所有有观察者（在本例中只有一个`Customer`）有关新产品（'iPhone 15'）的信息。该`customer.update()`方法被调用，并将新产品记录到控制台。

# **4. 发布者/订阅者模式**

发布者/订阅者模式是一种消息传递模式，其中消息的发送者（称为发布者）不会对消息进行编程以将其直接发送给特定接收者（称为订阅者）。相反，发布的消息被分类到各个类别中，而发布者并不知道订阅者的身份。这是处理事件驱动编程的有效方法。

在发布者/订阅者模式中，发布者不直接与订阅者通信。相反，这些消息被分类并通过消息总线发送给订阅者。这可以在复杂的系统中提供更大的灵活性和可扩展性。

## 真实案例

让我们考虑一个聊天组应用程序。 Pub/Sub 系统可以有效地处理消息的发送和接收：

### 订阅事件

当打开聊天窗口时，它会订阅一个事件，例如 `newMessage`。

这是通过 `pubsub.subscribe("newMessage", callback)`来实现的。当新消息发布到事件时，将执行回调函数`newMessage`。在本例中，回调会记录新消息并更新聊天 UI。

```javascript
const messageData = { user: "User A", text: "This is what the user had typed" };

pubsub.publish("newMessage", messageData);
```

通过这种方式，发布/订阅系统可以将聊天窗口（订阅者）和消息发送者（发布者）分离开来。聊天窗口不需要知道谁发送了消息，它们只需要知道收到消息后该做什么。

类似地，消息发送者不需要知道谁会收到消息；他们只需要将消息发送到正确的主题。

### 如何建立 pub sub

要构建 PubSub 系统，我们需要维护事件或“主题”及其各自订阅者的记录。这可以通过一个简单的 JavaScript 对象来完成。发布新消息时，我们会查找相关订阅者并执行其回调函数。这允许动态且灵活的系统，可以在运行时添加或删除发布者和订阅者。

```javascript
class PubSub {
        static events = {}; // It has the an empty list of events

    // The subscribe method takes an event name and a callback function
    subscribe(eventName, callback) {
        if (!this.events[eventName]) {
                // If the event doesn't exist yet, initialize it as an empty array
            this.events[eventName] = [];
        }

        // Push the callback function into the array of callbacks for the given event
        this.events[eventName].push(callback);
    }

    // The publish method takes an event name and data
    publish(event, data) {
        // If the event doesn't exist, or there's no subscribers for this event, return
        if (!this.events[event]) {
            return;
        }

        // For each subscriber of this event, call the callback function with the provided data
        this.events[event].forEach((callback) => {
            callback(data);
        });
    }
}
```

该类`PubSub`具有一个静态`events`属性，它是一个将存储所有事件（或主题）及其相应订阅者（回调函数）的对象。

该`subscribe`方法用于为给定事件注册新订阅者。它以事件名称和回调函数作为参数。如果事件尚不存在，它会将其初始化为对象中的空数组`events`。然后，它将回调函数添加到给定事件的回调数组中。

该`publish`方法用于向事件发布新数据。它以事件名称和要发布的数据作为参数。如果事件不存在或此事件没有订阅者，则它仅返回而不执行任何操作。如果有订阅者，则它会调用每个订阅者的回调函数，并将发布的数据作为参数传递。

在聊天应用程序的上下文中，该`subscribe`方法将用于注册一个应该接收新消息的新聊天窗口，并且该`publish`方法将用于向所有已订阅接收新消息的聊天窗口发送新消息。

# 5. 适配器模式

适配器模式是一种结构设计模式，允许具有不兼容接口的对象进行协作。当想让现有类与其他类协同工作而不修改其源代码时，通常会使用这种模式。当现有类的接口与所需的接口不匹配时，此模式特别有用。

## 真实案例

让我们考虑一个现实生活中的例子。你的任务是将第三方视频播放器集成到你的应用程序中。但是，视频播放器的功能不同，并且方法接口与应用程序所期望的不同。在这种情况下，可以使用适配器模式围绕视频播放器创建一个包装器类，使第三方代码与现有的应用程序代码兼容。

以下是在这种情况下要使用的代码：

```javascript
// Adapter class
class VideoPlayerAdapter {
    constructor() {
        this.externalPlayer = new ThirdPartyVideoPlayer({
            // some configuration
        });
    }
    play() {
        const video = this.externalPlayer.getVideo();
        this.externalPlayer.playVideo(video, {
            // additional parameters
        });
    }
}// Your application code
class Application {
    constructor() {
        this.videoPlayer = new VideoPlayerAdapter();
    }
    start() {
        // Play video using your application code
        this.videoPlayer.play();
    }
}
```

让我们分解一下上面的代码：

1. `ThirdPartyVideoPlayer`是你的应用程序想要使用的假设外部库。但是，其接口可能与应用程序不兼容。
2. `VideoPlayerAdapter`是适配器类。它包装了`ThirdPartyVideoPlayer`。适配器的接口与您的应用程序兼容。当调用适配器的`play()`方法时，它会在内部调用必要方法`ThirdPartyVideoPlayer`。
3. `Application`是应用程序代码。它创建 的一个实例`VideoPlayerAdapter`并将其用作常规视频播放器。当它调用适配器上的方法`play()`时，适配器会将其转换为对`ThirdPartyVideoPlayer`的适当调用。

这样，`Application`类就不需要知道任何`ThirdPartyVideoPlayer`工作原理。

# 6.复合模式

复合模式是一种结构设计模式，它将对象组织成树形结构，以表示整体-部分层次结构。它可以帮助以相同的方式处理单个对象和对象组。此模式非常适合处理树形结构，例如用户界面或组织层次结构。

在 JavaScript 中，复合模式可以通过创建一个定义简单对象和复合对象的通用接口的基组件类来实现。“叶”对象表示组合中的最终对象，而复合对象可以包含其他叶或复合对象。

## 真实案例

开发人员面临的一个常见挑战是管理带有嵌套元素的复杂 UI，例如带有子菜单的菜单或带有目录和文件的文件系统。如果没有结构化方法，代码可能会变得难以管理和扩展，从而导致潜在的错误和效率低下。

这就是复合模式发挥作用的地方。通过统一处理单个 UI 元素和元素组，开发人员可以简化嵌套组件的管理。例如，考虑一个菜单系统，其中我们有两种类型的组件：`MenuItem`和`Menu`。`MenuItem`表示单个菜单项，而`Menu`可以包含多个`MenuItem`对象和其他`Menu`对象。`MenuItem`和`Menu`都实现一个通用接口，允许主菜单统一处理它们。这种模式不仅使代码更易于管理，而且还增强了其可扩展性。

为了更好地理解组合模式，让我们使用 JavaScript 实现一个简单的示例。

```javascript
// Component interface
class MenuComponent {
  add(component) {}
  remove(component) {}
  getChild(index) {}
  display() {}
}// Leaf object
class MenuItem extends MenuComponent {
  constructor(name) {
    super();
    this.name = name;
  }display() {
    console.log(this.name);
  }
}// Composite object
class Menu extends MenuComponent {
  constructor(name) {
    super();
    this.name = name;
    this.children = [];
  }
  add(component) {
    this.children.push(component);
  }
  remove(component) {
    this.children = this.children.filter(child => child !== component);
  }
  getChild(index) {
    return this.children[index];
  }
  display() {
    console.log(this.name);
    this.children.forEach(child => child.display());
  }
}// Client code
const mainMenu = new Menu('Main Menu');
const menuItem1 = new MenuItem('Item 1');
const menuItem2 = new MenuItem('Item 2');
const subMenu = new Menu('Sub Menu');
const subMenuItem1 = new MenuItem('Sub Item 1');mainMenu.add(menuItem1);
mainMenu.add(menuItem2);
mainMenu.add(subMenu);
subMenu.add(subMenuItem1);// Display the menu structure
mainMenu.display();
```

让我们分解一下代码：

**MenuComponent：**这是基类，用作叶对象和复合对象的接口。它定义了常用方法，如`add`、`remove`、`getChild`和`display`。

**MenuItem：**该类表示复合模式中的叶对象。它扩展`MenuComponent`并实现了`display`输出其名称的方法。

**菜单：**此类表示可包含子项（`MenuItem`包括其他`Menu`对象）的复合对象。它扩展`MenuComponent`并实现添加、移除、检索和显示其子项的方法。

**客户端代码：**示例创建了一个菜单结构，其中包含主菜单、菜单项和子菜单。然后，它将菜单项添加到主菜单和子菜单中，最后显示整个结构。

# 7. 建造者模式

建造者模式是一种创建型设计模式，允许开发人员逐步构建复杂对象。它将对象的构建与其表示分离，使相同的构建过程能够创建不同的表示。

在 JavaScript 中，这种模式在处理需要大量可选参数的对象或构造过程涉及多个步骤时特别有用。通过使用建造者模式，开发人员可以通过链接方法调用来设置属性，从而创建更易读、更易于维护的代码，并最终以受控的方式构建所需的对象。

## 真实案例

想象一下，一位前端开发人员负责为 Web 应用程序创建复杂的仪表板。挑战在于允许用户使用各种小部件（例如图表、表格和通知）自定义仪表板。每个小部件都需要不同的设置，例如数据源、显示选项和刷新间隔。通过单个构造函数或多个设置器管理所有这些选项可能会变得繁琐且容易出错。

构建器模式通过使用`DashboardBuilder`类提供了一种优雅的解决方案，开发人员可以链接方法调用来设置每个小部件的属性，确保在构建仪表板之前应用所有必要的配置。这种方法不仅使代码更具可读性和可维护性，而且还允许灵活地创建具有不同复杂程度的不同类型仪表板。

下面的代码举例说明了所讨论的场景。它展示了前端开发人员如何使用 Builder Pattern 来创建此仪表板：

```javascript
class DashboardBuilder {
  constructor() {
    this.widgets = [];
  }
  addChart(type, data) {
    this.widgets.push({ type: 'chart', chartType: type, data });
    return this;
  }
  addTable(data) {
    this.widgets.push({ type: 'table', data });
    return this;
  }
  addNotification(message) {
    this.widgets.push({ type: 'notification', message });
    return this;
  }
  build() {
    return new Dashboard(this);
  }
}
class Dashboard {
  constructor(builder) {
    this.widgets = builder.widgets;
  }
  render() {
    this.widgets.forEach(widget => {
      switch (widget.type) {
        case 'chart':
          console.log(`Rendering a ${widget.chartType} chart with data:`, widget.data);
          break;
        case 'table':
          console.log('Rendering a table with data:', widget.data);
          break;
        case 'notification':
          console.log('Showing notification:', widget.message);
          break;
        default:
          console.log('Unknown widget type');
      }
    });
  }
}// Usage
const dashboard = new DashboardBuilder()
  .addChart('line', [1, 2, 3])
  .addTable({ header: ['Name', 'Age'], rows: [['Alice', 30], ['Bob', 25]] })
  .addNotification('Welcome to your dashboard!')
  .build();dashboard.render();
```

此代码演示了如何实现用于创建可自定义仪表板的 Builder 模式。以下是主要组件的细分：

**1.****`DashboardBuilder`****类别：**

- 管理仪表板的构建
- 具有添加不同类型的小部件（图表、表格、通知）的方法
- 每个方法都返回`this`以允许方法链接
- 该`build()`方法创建并返回一个新的 Dashboard 实例

**2.仪表板类：**

- 代表最终产品
- `DashboardBuilder`在构造函数中获取一个实例
- 有一个`render()`显示小部件的方法

**3.使用示例：**

- 创建带有折线图、表格和通知的新仪表板
- 使用方法链以可读的方式添加小部件
- 调用`build()`来创建仪表板实例
- 最后，调用`render()`显示仪表板

该模式允许灵活且可读地创建具有各种小部件和配置的复杂仪表板对象。

# 适配器模式与外观模式的区别

虽然适配器模式和[外观模式](https://dev.to/superviz/design-pattern-2-facade-pattern-1dhl)看起来很相似，但它们有不同的用途，并且在不同的环境中使用：

1. **目的**
    - **适配器模式**

        适配器模式的主要目的是使两个不兼容的接口相互兼容。它允许使用具有不同接口的现有类，就好像它实现了不同的接口一样。

    - **外观模式**

        外观模式的主要目的是为复杂的子系统提供简化的接口。它隐藏了子系统的复杂性，并提供了更高级的接口，使子系统更易于使用。

2. **用法**
    - **适配器模式**

        当需要集成与应用程序中现有类或接口不匹配的新类或库时使用它。适配器模式是关于将特定接口适配到预期接口。

    - **外观模式**

        当想要简化与复杂子系统的交互时，可以使用该模式。外观模式提供了一种与系统交互的直接方法，隐藏了其复杂性。

3. **设计**
    - **适配器模式**

        通常涉及创建一个新类（适配器），该类实现客户端期望的接口并将调用转换为适配的类。

    - **外观模式**

        涉及创建一个外观类，为客户端提供简化的方法，通常聚合来自子系统的多种功能。

总之，虽然两种模式都提供了一种处理现有代码的方法，但适配器模式侧重于接口兼容性，而外观模式侧重于简化与复杂系统的交互。
