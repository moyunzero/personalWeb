当客户端与服务器交互时，有两种方式来处理：

- **无状态：**客户端在每个请求中都包含所有必要的数据，因此服务器不存储任何先前的信息。
- **有状态：**服务器保留一些来自以前请求的数据，使得未来的交互依赖于过去的状态。
> 在软件系统中，状态是指在请求过程中持续存在的任何数据，例如用户会话、购物车或身份验证详细信息。

无状态和有状态架构之间的选择会影响可扩展性、性能、复杂性和成本。


# **1. 有状态架构**


在有**状态架构**中，系统会记住多个请求中的客户端或进程数据（**状态**）。


一旦客户端连接，服务器就会保留某些详细信息（例如用户偏好、购物车内容或身份验证会话），因此客户端不需要在每次请求时重新发送所有内容。


有状态系统通常将状态数据存储在数据库或内存存储中。


![%E6%88%AA%E5%B1%8F2025-02-11_21.39.24.png](images/blog/2024-09-20-36ddf5c0/img-27fd1707fd.png)

> 例如：在线购物时，当您将商品添加到购物车时，网站会记住您的选择。如果您离开以浏览更多商品，然后返回购物车，您的商品仍在那里，等待您结账。

## **状态架构中的常见模式**


### **1. 粘性会话**


如果您使用**内存**会话存储（即，每个应用服务器在本地保存自己的会话），则可以为“粘性会话”配置负载均衡器。


这意味着：一旦将客户端分配给**服务器 A** ，来自该客户端的所有后续请求都将路由到**服务器 A** 。


![%E6%88%AA%E5%B1%8F2025-02-11_21.40.29.png](images/blog/2024-09-20-36ddf5c0/img-9ddd6178e7.png)

> 权衡：如果服务器 A 出现故障，则用户的会话数据将丢失或强制用户重新登录。粘性会话在扩展时也不太灵活，因为您无法无缝地将用户流量重新分配到其他服务器。

### **2. 集中式会话存储**


更为强大的方法是将会话数据存储在**集中式**或**分布式**存储（例如 Redis）中。


![%E6%88%AA%E5%B1%8F2025-02-11_21.41.16.png](images/blog/2024-09-20-36ddf5c0/img-0f71be25a2.png)


这允许：

- **共享访问**：所有服务器都可以访问和更新任何用户的会话数据。任何服务器都可以处理任何请求，因为会话数据不与特定服务器的内存绑定。
> 权衡：您需要引入网络开销并依赖外部存储。如果集中式存储发生故障，除非您有后备策略，否则您将丢失会话数据。

## **优点：**

- **个性化体验：**状态系统可以提供高度定制的交互，因为它们可以记住用户的偏好和过去的行为。
- **上下文连续性：**即使用户断开连接并重新连接，他们也可以无缝地恢复上次中断的活动。
- **减少往返：**由于服务器已经拥有必要的数据，因此某些操作可以更快。

## **挑战：**

- **可扩展性：**由于每个服务器都需要跟踪特定的会话，因此维护大量用户的状态可能会耗费大量资源并且很复杂。
- **复杂性：**跨多个服务器管理和同步状态（如果需要）会带来额外的挑战。
- **故障点：**如果保存用户状态的服务器发生故障，则他们的会话数据可能会丢失。

## **示例用例**

- **电子商务购物车**——存储购物车内容和用户在多次交互中的偏好，即使用户离开后又返回。
- **视频流服务（Netflix、YouTube）** ——记住用户观看进度、推荐和会话数据，以获得无缝体验。
- **消息应用程序（WhatsApp、Slack）** ——维护活跃的用户会话和消息历史记录，以实现实时通信。

# **2.无状态架构**


在**无状态**架构中，服务器**不会**在各个请求之间保留客户端特定的数据。

- 每个请求都被视为**独立的**，不记得以前的交互。
- 每个请求必须包含处理**所需的所有必要信息**。
- 一旦服务器响应，它会丢弃用于该请求的**任何临时数据**。

![%E6%88%AA%E5%B1%8F2025-02-11_21.43.05.png](images/blog/2024-09-20-36ddf5c0/img-df151d6e1e.png)

> 示例：大多数RESTful API都遵循无状态设计。例如，当您从公共 API 请求天气数据时，您必须在每个请求中提供所有必需的详细信息（例如位置）。服务器处理它，发送响应，然后忘记交互。

## **无状态架构中的常见模式**


### **1.基于令牌的身份验证（JWT）**


实现无状态的一个非常流行的方法是通过令牌，特别是**JWT** （JSON Web 令牌）：

1. **客户端认证一次**：用户首次使用凭证（用户名/密码）登录，服务器发出签名的**JWT** 。
2. **后续请求**：客户端在每个请求中包含 JWT 令牌（例如， `Authorization: Bearer <token>`标头）。
3. **验证**：服务器验证令牌的签名和任何嵌入的声明（例如用户 ID、有效期）。
4. **无服务器端存储**：服务器不需要**存储**会话数据；它只需在每次请求时验证令牌。
> 许多 API（包括基于 OAuth 的身份验证系统）都使用 JWT 来实现无状态、可扩展的身份验证。

### **2. 幂等 API**


无状态架构受益于**幂等操作**，确保重复的请求产生相同的结果。这可以防止由于网络重试或客户端错误而导致的不一致。


**示例：**具有相同有效负载的 PUT /users/123 请求始终`PUT /users/123`**用户**的数据，但不会创建重复项。

> 幂等 API 确保一致性和可靠性，特别是在可能自动重试请求的分布式系统中。

## **优点：**

- **可扩展性：**无状态系统本质上更容易水平扩展。可以轻松添加新服务器，因为它们不需要维护任何特定的用户会话。
- **简单性：**由于服务器不跟踪状态，因此架构通常更简单且更易于管理。
- **弹性：**单个服务器的故障不会中断用户会话，因为数据不与特定服务器绑定。
- **降低内存占用：**由于服务器上没有存储会话数据，因此您可以释放原本为会话管理保留的内存。
- **更容易缓存响应：**由于请求是自包含的，缓存层（如 CDN）可以更轻松地存储和提供响应。

## **挑战：**

- **较少上下文：**无状态系统如果不付出额外努力（例如使用 cookie 或令牌），就无法提供与有状态系统相同级别的个性化或上下文感知。
- **客户端复杂性：**客户端必须跟踪身份验证令牌或相关数据。如果令牌丢失，则必须重新进行身份验证。
- **大有效载荷：**每个请求都需要携带所有必需的信息，这可能会导致更大的有效载荷。

## **示例用例**

1. **微服务架构：**每个服务独立处理请求，依赖外部数据库或缓存，而不是维护会话数据。
2. **公共 API（REST、GraphQL）：**客户端在每次请求时都会发送令牌，从而无需服务器端会话。
3. **移动应用程序：**令牌安全地存储在设备上，并随每次请求一起发送以验证用户身份。
4. **CDN 和缓存层：**无状态端点使缓存更容易，因为响应仅取决于请求参数，而不是存储的会话数据。CDN 可以缓存并处理重复的请求，提高性能并减少后端负载。

# **选择正确的方法**


在有状态和无状态架构之间进行选择时，没有一刀切的答案。


最佳选择取决于您的应用程序的需求、可扩展性目标和用户体验期望。


### **何时选择状态架构**


当**用户环境和连续性**至关重要时，有状态的系统是理想的选择。


如果您的应用程序出现以下情况，请考虑采用状态方法：

- 需要个性化（例如用户偏好、会话历史记录）
- 需要实时交互（例如聊天应用程序、多人游戏）
- 管理多步骤工作流程（例如网上银行交易、结账流程）
- 必须保留身份验证会话以确保安全性和方便性
> 示例：电子商务应用中的购物车应该持久保存，这样用户不必在刷新页面后重新添加商品。

### **何时选择无状态架构**


当**可扩展性、简单性和弹性**是首要考虑因素时，无状态系统效果最佳。


如果您的应用程序符合以下情况，请使用无状态方法：

- 处理大量请求并需要有效扩展
- 不需要在请求之间存储客户端特定数据
- 需要快速、分布式处理，不依赖服务器
- 必须确保可靠性和故障转移准备
> 示例：天气 API 不需要记住之前的请求。每个查询都包含位置，并且响应是独立处理的。

### **混合方法：两全其美**


许多现代应用程序**混合了**有状态和无状态组件以实现灵活性。


这种混合方法可以实现：

- 核心功能采用无状态 API，确保高可扩展性
- 有状态会话用于个性化，改善用户体验
- 外部会话存储（例如 Redis）用于管理状态，同时保持应用服务器无状态
> 示例：视频流平台（例如 Netflix）使用无状态后端进行流式传输，但保留有状态的用户会话以跟踪观看历史和推荐。
